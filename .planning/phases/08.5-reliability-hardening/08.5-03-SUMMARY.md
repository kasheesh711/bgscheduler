# Phase 08.5 Plan 03: REL-03 Identity Collision Detection Summary

Identity resolver in `src/lib/normalization/identity.ts` now emits an `identity_collision:` data_issue when a nickname maps to 3+ teachers, or to 2 teachers that are NOT a legitimate online/offline pair — eliminating silent triple-merge of unrelated tutors into one phantom identity group.

## What Shipped

**Behavior change in `resolveIdentities` for-loop (lines 146–172):**

After `groups.push(group)` and before the `resolved.add(...)` iteration, the resolver now classifies each canonicalKey grouping:

| Entries | Online | Offline | Disposition | Issue Emitted |
|---------|--------|---------|-------------|---------------|
| 1 | any | any | solo merge (preserved) | none |
| 2 | 1 | 1 | legitimate pair (preserved) | none |
| 2 | 0 | 2 | collision | `identity_collision:` |
| 2 | 2 | 0 | collision | `identity_collision:` |
| ≥3 | any | any | collision | `identity_collision:` |

The collision issue carries:
- `type: "alias"` (existing pgEnum value reused per RESEARCH A1)
- `entityType: "tutor_identity_group"`
- `entityId: group.canonicalKey`, `entityName: group.displayName`
- `message: 'identity_collision: nickname "<X>" matched N teachers (P online, Q offline) — needs manual disambiguation. Members: <comma-joined display names>'`

The group itself is still pushed to `groups[]` so its members appear in Needs Review (fail-closed). The orchestrator at `src/lib/sync/orchestrator.ts:86-97` persists the issue with hardcoded `severity: "critical"` — no orchestrator change required.
