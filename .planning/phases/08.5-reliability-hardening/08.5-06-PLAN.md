<objective>
Replace the silent `.catch(() => {})` at `src/lib/sync/orchestrator.ts:524` with a logged catch that emits to `console.error` with the syncRunId, the primary error message, and the cleanup error itself. This restores operational visibility when the DB is fully unreachable and the cleanup write to `sync_runs` also fails.

Purpose: Today, if the database becomes unreachable mid-sync, the outer `try/catch` at orchestrator.ts:512 catches the primary error, attempts to mark the sync run as `failed`, and silently swallows any failure of THAT cleanup write via `.catch(() => {})`. Operators see a sync run stuck in `running` state on the `/data-health` page with no log evidence of what actually happened. Logging the cleanup error closes the gap.
Output: Single-line replacement of the empty catch with a logged catch.

```typescript
    if (syncRunId) {
      await db
        .update(schema.syncRuns)
        .set({
          status: "failed",
          finishedAt: new Date(),
          errorSummary: errorMessage,
        })
        .where(eq(schema.syncRuns.id, syncRunId))
        .catch((cleanupErr) => {
          // REL-06: surface the cleanup failure in Vercel logs so an operator
          // can see WHY the sync_runs row is stuck in `running` state. We
          // still swallow (don't re-throw) because the primary errorMessage
          // is what matters for the SyncResult — masking it with a cleanup
          // error would be worse than the current behavior.
          const cleanupMsg =
            cleanupErr instanceof Error ? cleanupErr.message : String(cleanupErr);
          console.error(
            `[sync-orchestrator] cleanup failed for syncRunId=${syncRunId} after primary error "${errorMessage}":`,
            cleanupMsg,
          );
        });
    }
```

Important:
- Do NOT re-throw inside the catch — the primary `errorMessage` must remain the canonical failure cause.
- The log line uses the literal prefix `[sync-orchestrator] cleanup failed` for easy log-search filtering.
- Use `console.error` (not `console.log`) to ensure the entry hits Vercel's error log stream.
- `errorMessage` is already in scope from line 513 (`const errorMessage = err instanceof Error ? err.message : String(err);`) — no new variable needed.
- Do NOT touch the rest of the catch block or the `return` statement that follows.
