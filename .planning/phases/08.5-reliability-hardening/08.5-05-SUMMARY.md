# Phase 08.5 Plan 05: REL-05 Status-Code-Aware Retry Policy — Summary

**One-liner:** Replaced the indiscriminate "retry any error" loop in `WiseClient.fetchWithRetry` with a status-code-aware policy: permanent 4xx (401/403/404/422) now throw on the first response, while transient errors (5xx, 408, 429, network) continue to retry with the same 1s/2s/4s exponential backoff — pinned by 5 new unit tests.

## What shipped

### Task 1 — Add retry-policy unit tests (RED) — `88d45d9`

Added a new `describe("WiseClient — REL-05 status-code-aware retry policy")` block in `src/lib/wise/__tests__/client.test.ts` with 5 cases:

| # | Case | Status under OLD code | Status under NEW code |
|---|------|-----------------------|----------------------|
| 1 | 401 (permanent 4xx) → no retry, fetch called 1× | FAIL (called 4×) | PASS |
| 2 | 404 (permanent 4xx) → no retry, fetch called 1× | FAIL (called 4×) | PASS |
| 3 | 500 (transient 5xx) → retry maxRetries times then throw, fetch called 4× | PASS | PASS |
| 4 | 429 (rate limit) → retry, second call returns 200 | PASS | PASS |
| 5 | Network error (TypeError) → retry, second call returns 200 | PASS | PASS |

Each test wraps in `vi.useFakeTimers()` so the 1s/2s/4s backoffs do not stretch the suite. The 500 test attaches `expect(...).rejects.toThrow(...)` BEFORE `vi.runAllTimersAsync()` to keep vitest from flagging the rejection as unhandled while fake timers advance.

RED state captured: tests 1 and 2 failed against the original code with the diagnostic `expected "vi.fn()" to be called 1 times, but got 4 times` — proving the original code retries 401/404 indiscriminately (`1 initial + maxRetries = 4`).

### Task 2 — Implement RETRYABLE_STATUS_CODES (GREEN) — `383d166`

Modified `src/lib/wise/client.ts`:

**Added** at the top of the `WiseClient` class:

```typescript
private static readonly RETRYABLE_STATUS_CODES: ReadonlySet<number> = new Set([
  408, // Request Timeout
  429, // Too Many Requests
  500, // Internal Server Error
  502, // Bad Gateway
  503, // Service Unavailable
  504, // Gateway Timeout
]);
```

**Replaced** the single try/catch in `fetchWithRetry` with three explicit paths:

1. **Network failure path** — `try { response = await fetch(...) } catch (networkErr)`. Retries with 1s/2s/4s backoff, throws after maxRetries.
2. **Success path** — `if (response.ok) return await response.json()`.
3. **Status-aware error path** — reads `response.text()` once, then:
   - **Permanent**: `if (!WiseClient.RETRYABLE_STATUS_CODES.has(response.status)) throw new Error(\`Wise API ${response.status}: ${text} (${url})\`)` — fail fast.
   - **Transient**: retry with same backoff; throw same error after maxRetries.

Public `get<T>` / `post<T>` API and concurrency limiter unchanged.
