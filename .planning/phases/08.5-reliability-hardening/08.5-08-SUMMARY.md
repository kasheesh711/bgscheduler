# Phase 08.5 Plan 08: Timezone Idiom Unification (REL-08) Summary

Replaced 6 occurrences of the `new Date(new Date().toLocaleString("en-US", { timeZone: "Asia/Bangkok" }))` idiom with `toZonedTime(new Date(), TIMEZONE)` from `date-fns-tz` across 4 production files, and added 3 round-trip lock-in tests in `timezone.test.ts` to pin the contract.

## Why It Matters

Per `08.5-RESEARCH.md` REL-08 deep dive (lines 661-708) and `CONCERNS.md` (lines 56-62): the previous idiom

```ts
new Date(new Date().toLocaleString("en-US", { timeZone: "Asia/Bangkok" }))
```

produced a Date whose UTC fields *appeared* to be Bangkok-local but was interpreted by all other Date code as UTC. Subsequent `.getDay()` / `.getHours()` reads accidentally worked because the internal UTC fields matched the desired Bangkok wallclock — but the resulting Date is **not a valid instant**. Any future code path using `.getTime()` or comparing instants would silently drift.

`toZonedTime(new Date(), TIMEZONE)` produces a properly-formed Date whose component-method outputs are identical at the consumer level for these read paths, eliminating the silent-drift hazard. The 3 lock-in tests pin the observable contract so a future revert to the unsafe idiom would fail tests.
