# Phase 08.5 Plan 04: REL-04 Multi-Day Leave Overlap Refactor Summary

Refactored `hasRecurringLeaveConflict` at `src/lib/search/engine.ts` so the multi-day vs single-day distinction is explicit, the full-day-block assumption is documented in JSDoc, and the misleading use of `leaveStart.getHours()/getMinutes()` outside the single-day branch is eliminated. Behavior is unchanged from production; 2 lock-in tests now gate against future regressions.

## What Shipped

### Task 1 — Lock-in tests (commit `baacbe2`)

Added a new `describe("executeSearch — REL-04 leave overlap (recurring mode)")` block with two `it()` cases:

1. **Multi-day leave (Mon 14:00 → Wed 10:00) blocks Tue 14:00-16:00 slot** — verifies the documented full-day-block semantics for multi-day leaves.
2. **Single-day leave (Mon 14:00 → Mon 16:00) blocks Mon 15:00 but NOT Tue 14:00 or Mon 13:00** — verifies that minute-of-day overlap math applies only on the leave's own weekday (3 sub-cases: A within-leave/blocked, B before-leave-same-day/free, C different-weekday/free).

The tests reuse the existing `makeTutor` and `makeIndex` fixture helpers, plus a small `makeTutorWithLeave` helper that ensures the tutor has Mon and Tue availability windows so the leave is the only thing that can block.

Both tests pass against the pre-refactor code (this is intentional — they lock in the documented assumption ahead of Task 2's refactor, not RED-then-GREEN).

### Task 2 — Refactor `hasRecurringLeaveConflict` (commit `e55e4a8`)

Replaced the loop-with-shadowed-variable form with explicit branching:

- The `isMultiDay` calculation moves to the top of the per-leave loop body.
- The `if (isMultiDay) { … continue; }` branch walks each calendar day in `[leaveStart, leaveEnd]` and blocks on any weekday match — no minute-of-day math involved (every touched weekday is blocked in full).
- The single-day branch uses `leaveStart.getDay()`, `leaveStart.getHours()`, `leaveEnd.getHours()` directly. By definition `leaveStart` and `leaveEnd` are on the same calendar day in this branch, so the minute-of-day math is correct.
- Added a JSDoc paragraph above the function documenting the full-day-block assumption per RESEARCH Pitfall 3 / REL-04 deep dive.
- Function signature is unchanged. No callers needed updating.

Both lock-in tests + all 141 prior tests pass against the refactored code.

## Threat Model Outcomes

- **T-08.5-08 (Tampering — false availability)** mitigated. The refactor preserves the full-day-block semantics for multi-day leaves and the lock-in tests prevent future regressions where someone "fixes" the misleading variable use into something behaviorally different.
- **T-08.5-09 (Repudiation — incorrect search results)** accepted as documented. Behavior is unchanged from production.
- **Residual risk** documented in PLAN: a leave shape "leave 23:00 Mon to 01:00 Tue" (2h total span) is technically multi-day under the literal `> 24h` threshold check — actually FALSE (2h is not > 24h), so this would fall to the single-day branch and use `leaveStart.getDay() = 1` (Monday) for the weekday check, NOT blocking Tuesday. This is consistent with the prior code's behavior; tests were not extended to cover the threshold-edge case as it falls outside this plan's scope and is not a new risk introduced by the refactor.
