<objective>
Replace the non-atomic two-UPDATE snapshot promotion at `src/lib/sync/orchestrator.ts:469-484` with a single SQL UPDATE statement so PostgreSQL MVCC guarantees that no concurrent reader ever sees zero rows where `active = true`.

Purpose: Today, a network failure or process termination between the deactivate and activate UPDATEs leaves the system with ZERO active snapshots — every subsequent `ensureIndex` / `buildIndex` call throws `"No active snapshot found"` (HTTP 500). The application is unavailable until the next successful sync.
Output: A single, atomic, bounded UPDATE statement at the promotion site.
</objective>

**Step 2 — Replace lines 469-484 (the entire `if (shouldPromote) { ... }` block) with the atomic single-statement form:**

The current code:
```typescript
    if (shouldPromote) {
      // Atomic promotion: deactivate old, activate new
      await db
        .update(schema.snapshots)
        .set({ active: false })
        .where(
          and(eq(schema.snapshots.active, true), sql`${schema.snapshots.id} != ${snapshotId}`)
        );

      await db
        .update(schema.snapshots)
        .set({ active: true })
        .where(eq(schema.snapshots.id, snapshotId));

      promotedSnapshotId = snapshotId;
    }
```

Becomes:
```typescript
    if (shouldPromote) {
      // Atomic promotion via a single UPDATE: PostgreSQL MVCC + the row-level
      // lock held for the duration of one statement guarantee that concurrent
      // readers see either the prior-active row or the new-active row — never
      // a moment with zero matches on `active = true`. The bounded WHERE
      // restricts the rewrite to (a) the previous active row(s) and (b) the
      // candidate snapshot, avoiding a full-table rewrite per promote.
      // Replaces the prior two-UPDATE sequence (REL-01).
      await db
        .update(schema.snapshots)
        .set({
          active: sql`(${schema.snapshots.id} = ${snapshotId})`,
        })
        .where(
          or(
            eq(schema.snapshots.active, true),
            eq(schema.snapshots.id, snapshotId),
          ),
        );

      promotedSnapshotId = snapshotId;
    }
```

The `sql\`(${schema.snapshots.id} = ${snapshotId})\`` expression is parameterized — Drizzle binds `snapshotId` as a query parameter, NOT string-interpolated. PostgreSQL evaluates the boolean expression `(id = $newId)` per-row inside the single UPDATE, setting `active=true` for the matching row and `active=false` for the previously-active row(s). The bounded `WHERE` clause keeps this from being a full-table sweep.

**Do NOT touch** the diff-hook code (orchestrator.ts:396, past-sessions-diff-hook.ts:60-62). REL-01 scope is JUST the snapshot promotion — see Pitfall 6 in the research.
