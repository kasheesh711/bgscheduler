## Summary

Phase 8.5 is a **remediation phase**, not a feature build. Eight defects were identified in the 2026-04-29 codebase audit (CONCERNS.md). For each, the fix approach is short and well-understood, with the open questions limited to: (1) which Neon driver to use for atomic snapshot promotion, (2) whether the `ensureIndex` race is actually live in the V8 model, and (3) the day-cursor minute-of-day computation for multi-day leaves.

Independent verification this session resolved every research question without ambiguity:

- **REL-01 atomic promotion:** A single SQL statement updating multiple rows is atomic in PostgreSQL via MVCC. The Neon HTTP driver also supports `sql.transaction([...])` (non-interactive). **Either approach works** — recommended fix is the single-statement form (simpler; no driver swap; no migration).
- **REL-02 ensureIndex race:** The race IS live. Between `getCurrentIndex()` (line 282) and `setBuildingPromise(p)` (line 301) there is an `await` on the active-snapshot DB lookup (lines 285-289), which yields to the microtask queue. A second concurrent caller can interleave during that yield. The fix is the synchronous singleton-promise pattern: assign `setBuildingPromise(p)` BEFORE any await, treating any duplicate caller as a no-op.
- **REL-04 multi-day leaves:** The current code already short-circuits multi-day leaves to "block the whole day" (line 259-262). The bug only manifests for **single-day leaves whose start-day is NOT the requested weekday** — and that case is impossible (a single-day leave only spans one weekday by definition). Recommendation: **option (b)** — keep the full-day-block behavior, document the assumption, fix the misleading variable naming.
- **REL-05 retry policy:** Standard retryable codes are `408, 429, 500, 502, 503, 504` and network errors. The current code retries on **everything**, including 401/404/422 (permanent errors).
- **REL-07 CRON_SECRET:** Cloudflare Workers' canonical pattern is `lengthsMatch ? timingSafeEqual(a, b) : !timingSafeEqual(a, a)`. Same pattern works in Node.js with `crypto.timingSafeEqual` against `Buffer.from(...)`.
- **REL-08 timezone idiom:** `toZonedTime(new Date(), "Asia/Bangkok")` is a behavioral drop-in replacement at every call site. All 6 occurrences (4 cited + 2 internal duplicates within calendar-grid and week-overview) consume the result via `.getHours()`, `.getMinutes()`, `.getDay()` — exactly what `toZonedTime` produces.

**Primary recommendation:** Ship as 4 parallel waves. REL-01/02/03/04/06 are orchestrator+search-internals (single owner, single test file). REL-05 is wise-client. REL-07 is API route. REL-08 is timezone idiom (touches 5 files but each touch is identical 3-line edit). No new dependencies. No driver migration. All fixes within Drizzle + Neon HTTP + Node.js stdlib.

### Pattern 1: Single-Statement Atomic UPDATE (REL-01)

**What:** Replace two-step deactivate+activate with one statement using a CASE expression.

**When to use:** Whenever multiple rows need a coordinated boolean flag flip and the driver does NOT support interactive transactions.

**Source:** `[CITED: postgresql.org/docs/current/explicit-locking.html]` — UPDATE acquires ROW EXCLUSIVE; row-level locks are held for the duration of the statement; MVCC ensures readers see either the pre-state or post-state, never an in-flight mix.

**Example:**

```typescript
// REPLACES src/lib/sync/orchestrator.ts:471-481 (the two-UPDATE sequence)

// Before (non-atomic — window where 0 rows have active=true):
//   await db.update(snapshots).set({ active: false }).where(and(eq(active, true), sql`id != ${snapshotId}`));
//   await db.update(snapshots).set({ active: true }).where(eq(snapshots.id, snapshotId));

// After (atomic, single statement):
await db
  .update(schema.snapshots)
  .set({
    active: sql`(${schema.snapshots.id} = ${snapshotId})`,
  })
  // Bound the UPDATE to the affected snapshots only — avoids a full-table
  // rewrite on each promote (tens of MB of dead tuples after a year).
  .where(
    or(
      eq(schema.snapshots.active, true),
      eq(schema.snapshots.id, snapshotId),
    ),
  );
```

PostgreSQL processes the entire statement under a single MVCC snapshot. Concurrent readers see either the old active row or the new active row — never zero matches.

**Bound the WHERE clause** so we touch only the previous-active row(s) and the candidate, NOT every row in `snapshots`. After 30+ days of snapshots that's a meaningful difference.

### Pattern 2: Synchronous Singleton-Promise (REL-02)

**What:** Move the singleton-promise assignment to the synchronous prelude — before any `await`.

**When to use:** Lazy-init of an async resource where multiple concurrent callers must coalesce into one initialization.

**Source:** `[CITED: jonmellman.com/posts/singleton-promises]` — assigning `this.connectionPromise` synchronously guarantees subsequent callers reuse the same promise.

**Example:**

```typescript
// REPLACES src/lib/search/index.ts:281-305 (the existing ensureIndex)

export async function ensureIndex(db: Database): Promise<SearchIndex> {
  // Fast path: existing in-flight rebuild — return its promise (no DB hit).
  const inFlight = getBuildingPromise();
  if (inFlight) return inFlight;

  const cached = getCurrentIndex();

  // Synchronous build kickoff — assign the promise BEFORE the first await.
  // Concurrent callers checking getBuildingPromise() will now see the
  // assigned promise and return early.
  const buildAndCheck = async (): Promise<SearchIndex> => {
    if (cached) {
      const [activeSnapshot] = await db
        .select({ id: schema.snapshots.id })
        .from(schema.snapshots)
        .where(eq(schema.snapshots.active, true))
        .limit(1);
      if (activeSnapshot && activeSnapshot.id === cached.snapshotId) {
        return cached;
      }
    }
    return buildIndex(db);
  };

  const p = buildAndCheck().finally(() => setBuildingPromise(null));
  setBuildingPromise(p);
  return p;
}
```

Critical: `setBuildingPromise(p)` is called BEFORE `p` (or `buildAndCheck()`) hits its first await. The `getBuildingPromise() ? return` short-circuit at the top is now reachable for any concurrent caller.

### Anti-Patterns to Avoid

- **Driver swap to neon-serverless WebSocket just for atomicity:** Cold-start cost, additional ENV var, more code to migrate. Avoid — single-statement UPDATE achieves the same atomicity over HTTP.
- **`AsyncMutex` library or `p-limit`-style coordination for REL-02:** Stdlib singleton-promise pattern is sufficient and well-known. New dep is unjustified.
- **`crypto.timingSafeEqual` without length pre-check:** Throws on length mismatch. Must guard with `lengthsMatch ? equal(a,b) : !equal(a,a)` to avoid the throw AND avoid leaking length via timing.
- **`.catch(() => {})` that silently absorbs an error:** REL-06 specifically calls this out. Always log the error you decide not to throw.
- **Computing `leaveStart.getHours()*60 + leaveStart.getMinutes()` outside the day-cursor loop:** Variable naming suggests "start of leave" but the loop semantically iterates over each day in the leave range. Either compute fresh per-iteration (costly: another `getHours()` call per day) or document that multi-day leaves block the entire day.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic snapshot promotion | Custom 2-phase commit / file lock / lease | Single SQL UPDATE statement (PG MVCC guarantees) | Decades-proven; no extra moving parts |
| Constant-time string compare | Manual XOR loop / character-by-character compare | `crypto.timingSafeEqual` (Node stdlib) | Hand-rolled compares are notoriously hard to make actually constant-time across V8 |
| Coalesced async init | Custom AsyncMutex / Lock / Semaphore | Singleton-promise pattern (assign before await) | JS run-to-completion makes synchronous assignment a sufficient mutex |
| Timezone-aware "now" | `new Date(new Date().toLocaleString("en-US", {timeZone:...}))` | `toZonedTime(new Date(), TIMEZONE)` from `date-fns-tz` | The toLocaleString trick produces a Date whose internal UTC value lies; downstream date arithmetic accidentally works but breaks under DST or comparison boundaries |
| HTTP retry policy | "retry on any error" loop | Status-code-aware retry: 5xx + 408 + 429 + network only | 4xx (except 429) are permanent — retrying wastes budget and amplifies bad input |

**Key insight:** All five remediations replace hand-rolled or naive patterns with stdlib / driver-provided primitives. Zero new dependencies.

## Common Pitfalls

### Pitfall 1: Confusing "single SQL statement" with "single Drizzle call"

**What goes wrong:** Drizzle's fluent API may emit multiple SQL statements under the hood (e.g., when you use `.set()` with a subquery vs a `sql\`...\`` expression). The atomicity guarantee is at the **PostgreSQL statement** level, not the Drizzle method level.

**Why it happens:** ORMs sometimes prepare statements separately or use multi-step strategies internally for complex updates.

**How to avoid:** Use a `sql\`...\`` template literal for the SET expression so Drizzle compiles to literally one UPDATE statement. Verify by enabling Drizzle's query logger temporarily (or by reading the generated SQL via `.toSQL()`).

**Warning signs:** Two queries hitting the DB in pgAdmin or Neon's SQL Editor "recent queries" panel after a single Drizzle call.

### Pitfall 2: REL-02 fix that still defers the assignment to inside an async closure

**What goes wrong:** A naive "fix" wraps the existing code in `async () => { ... }` and assigns the promise immediately, but the inside still has the original race because the await still happens before the cached snapshot check completes.

**Why it happens:** People assume `setBuildingPromise(p)` is enough — but if the function body of `p` reads `getCurrentIndex()` and then awaits the DB before deciding to call `buildIndex`, two concurrent invocations can still both decide to call `buildIndex`.

**How to avoid:** Either (a) the synchronous prelude returns the in-flight promise unconditionally if one exists (current pattern, just moved earlier), or (b) accept that `buildIndex` itself is the coalescing surface — make `buildIndex` idempotent (check `getCurrentIndex().snapshotId === activeSnapshot.id` at the start before executing the parallel queries).

**Warning signs:** Two `buildIndex` calls show up in concurrent test under a `vi.spyOn(buildIndex)`.

### Pitfall 3: REL-04 single-day leave on a different weekday

**What goes wrong:** Someone reads "the current code uses leaveStart.getHours() regardless of which day the loop is on" and assumes that's a real bug. They write a fix that computes `current.getHours() * 60 + current.getMinutes()` — but `current` is a Date constructed via `new Date(leaveStart); current.setDate(current.getDate()+1)`, so `current.getHours()` returns the hours from the original `leaveStart`, NOT zero.

**Why it happens:** Confusing "iterate over days" with "midnight-cursor". JS `Date.setDate(n)` only changes the day-of-month — it preserves hours/minutes.

**How to avoid:** If we want fresh start/end minutes per day, the math must be: day 1 uses leaveStart's HH:MM; day N uses 00:00 to 23:59 (full block); last day uses 00:00 to leaveEnd's HH:MM. **This is exactly what "always treat multi-day leaves as full-day blocks" achieves** — and it's what the existing `isMultiDay` short-circuit already does.

**Recommendation:** Choose **option (b) from the requirement**: always treat multi-day leaves as full-day blocks. Document the assumption. Rename the misleading `leaveStartMin`/`leaveEndMin` variables to clarify they're only used for single-day leaves.

**Warning signs:** Test failures where a leave from "Mon 14:00 to Mon 16:00" (single day, single weekday) is incorrectly evaluated against a Tuesday slot.

### Pitfall 4: timingSafeEqual on differing-length buffers

**What goes wrong:** `crypto.timingSafeEqual(a, b)` THROWS a RangeError if `a.length !== b.length`. So even if you wrap it in try/catch, the throw itself takes a measurable amount of time (depending on stack unwind cost).

**Why it happens:** People treat it as a drop-in for `===` without reading the docs.

**How to avoid:** Use the Cloudflare-canonical guard: `lengthsMatch ? timingSafeEqual(a, b) : !timingSafeEqual(a, a)`. The "compare against self" path is always true (length matches itself); negating returns the right boolean while still running the constant-time path.

**Warning signs:** Uncaught RangeError: "Input buffers must have the same byte length" in production logs.

### Pitfall 5: REL-08 behavioral drift between toZonedTime and toLocaleString

**What goes wrong:** Someone replaces `new Date(new Date().toLocaleString("en-US", { timeZone: "Asia/Bangkok" }))` with `toZonedTime(new Date(), "Asia/Bangkok")` and a downstream `getDay()`/`getHours()` returns a different value, breaking a test.

**Why it doesn't actually happen here:** Both expressions produce a Date whose `.getHours()`, `.getMinutes()`, `.getDay()` return the same Bangkok-local values. The difference is in the underlying UTC instant — but no consumer reads it as a UTC instant. Verified via:
- `use-compare.ts:23-26` consumes via `bkk.getDay()` then `bkk.getFullYear()/getMonth()/getDate()`
- `week-overview.tsx:235-240` consumes via `bkk.getHours()*60 + bkk.getMinutes()`, `bkk.getDay()`, `bkk.toDateString()`
- `calendar-grid.tsx:67-71` same pattern as week-overview
- `compare/route.ts:28-31` consumes via `now.getDay()`, `now.getFullYear()/getMonth()/getDate()`

**How to avoid (proactive):** Add one assertion in the changed file's existing test (or new test): build a Date via the new helper, verify `.getDay()` and `.getHours()` round-trip to expected Bangkok values for a known UTC instant.

**Warning signs:** Today indicator jumps to wrong day after midnight (BKK local), or weekday labels in compare view are off-by-one.

### Pitfall 6: ATOMICITY scope of REL-01 fix vs. PAST-01 diff-hook ordering

**What goes wrong:** Someone reads "atomic promotion" and rewrites both the diff-hook insert AND the snapshot promotion as a single transaction. But the diff-hook explicitly runs BEFORE promotion (orchestrator.ts:396) and uses `ON CONFLICT DO NOTHING` for idempotency — the constraint that motivated the no-transaction acknowledgment at past-sessions-diff-hook.ts:60-62.

**Why it happens:** Conflating "I should make THIS atomic" with "everything should be atomic".

**How to avoid:** REL-01 scope is JUST the snapshot promotion (lines 469-484). The diff-hook's idempotency strategy is unchanged.

**Warning signs:** A REL-01 plan modifies past-sessions-diff-hook.ts, or removes ON CONFLICT DO NOTHING.

### REL-01 — Atomic snapshot promotion

**Cited site:** `src/lib/sync/orchestrator.ts:469-484` (the two `db.update(...)` calls inside the `if (shouldPromote)` block)
**Acknowledgment of HTTP no-transaction:** `src/lib/sync/past-sessions-diff-hook.ts:60-62` (in module-level docblock)

**Research question:** Does Neon HTTP guarantee atomicity for a single multi-row UPDATE?

**Answer:** **YES.** `[VERIFIED: postgresql.org/docs/current/explicit-locking.html]`. PostgreSQL UPDATE acquires a ROW EXCLUSIVE lock for the duration of the statement; row-level locks are held until end-of-statement; MVCC ensures readers see a consistent snapshot. **Additionally:** the Neon HTTP driver also supports `sql.transaction([...])` (non-interactive) per `[CITED: github.com/neondatabase/serverless]` and `[CITED: neon.com/docs/serverless/serverless-driver]` — so even multi-statement atomicity is available over HTTP without a driver swap. **The single-statement form is preferred** because it's one round-trip vs two.

**Recommended fix:**

```typescript
// orchestrator.ts:469-484 replacement
if (shouldPromote) {
  // Atomic promotion via single UPDATE: PostgreSQL guarantees row-level
  // locking for the duration of the statement and MVCC ensures concurrent
  // readers see either the pre-state or post-state, never an in-flight mix.
  // The WHERE clause bounds the rewrite to (a) the prior active row(s) and
  // (b) the candidate snapshot — avoiding a full-table rewrite per promote.
  await db
    .update(schema.snapshots)
    .set({
      active: sql`(${schema.snapshots.id} = ${snapshotId})`,
    })
    .where(
      or(
        eq(schema.snapshots.active, true),
        eq(schema.snapshots.id, snapshotId),
      ),
    );

  promotedSnapshotId = snapshotId;
}
```

**Tradeoffs of WebSocket alternative (rejected):**
- New ENV var (`DATABASE_URL_WS` or pool config) — operational burden
- Cold-start latency (WebSocket handshake adds ~80-200ms per cold function) — measurable in our 4m26s sync budget
- Code change at `src/lib/db/index.ts` `createDb()` — currently uses `neon(databaseUrl)` HTTP factory; would need switch to `Pool({ connectionString })` for transaction callbacks
- Loses the ergonomics of a single statement (would still need to write `BEGIN; UPDATE...; UPDATE...; COMMIT` as separate calls in `pool.transaction()` callback)

**Confidence:** HIGH. Single-statement atomicity is decades-proven PG behavior. No driver migration needed.

### REL-02 — Concurrent index rebuild coalescing

**Cited site:** `src/lib/search/index.ts:281-305` (`ensureIndex`)

**Research question:** Is the race actually live in V8's run-to-completion model, given the existing code at lines 281-305?

**Answer:** **YES, the race is live.** The existing code at line 282 calls `getCurrentIndex()` (sync), then at lines 285-289 awaits a DB query. THAT AWAIT yields to the microtask queue. A second concurrent caller can:
1. Enter `ensureIndex` while caller-1 is awaiting the DB lookup (line 285)
2. Read the same `cached` value (still set, but with stale snapshotId)
3. Await its own DB lookup (line 285)
4. Both callers proceed to line 297 (`if (!getBuildingPromise())`) — both see `null`
5. Both create a new build promise; the second `setBuildingPromise(p)` at line 301 overwrites the first; both `buildIndex(db)` calls run; one result is discarded

`[CITED: jonmellman.com/posts/singleton-promises]` — confirms this is the canonical async-init bug. The fix is the singleton-promise pattern: assign the promise BEFORE any await.

**Recommended fix:** See "Pattern 2: Synchronous Singleton-Promise" above. Smallest patch: 8-line rewrite of `ensureIndex`. No new dependency.

**Smallest valid patch (alternative):** Hoist the in-flight check to the very top, BEFORE the cached-snapshotId read:

```typescript
export async function ensureIndex(db: Database): Promise<SearchIndex> {
  const inFlight = getBuildingPromise();
  if (inFlight) return inFlight;

  // ... rest of existing logic ...

  // When kicking off a new build, assign synchronously before the first await:
  if (!getBuildingPromise()) {
    const p = (async () => { /* existing build logic */ })()
      .finally(() => setBuildingPromise(null));
    setBuildingPromise(p);
    return p;
  }
  return getBuildingPromise()!;
}
```

But the cleaner full rewrite (Pattern 2 above) is preferred — easier to test.

### REL-04 — Multi-day leave overlap

**Cited site:** `src/lib/search/engine.ts:240-269` (`hasRecurringLeaveConflict`)

**Research question:** Real-world bug trigger? Existing test coverage?

**Reading the existing code:** Lines 246-267 walk each day in `[leaveStart, leaveEnd]`. For each day matching the requested weekday:
- Compute `leaveStartMin = leaveStart.getHours() * 60 + leaveStart.getMinutes()` (line 255) — note: ALWAYS uses `leaveStart`, never `current`
- Compute `leaveEndMin = leaveEnd.getHours() * 60 + leaveEnd.getMinutes()` (line 256) — same issue
- If `isMultiDay` (>24h leave): block the whole day (`return true` regardless of minutes)
- Else: check `leaveStartMin < endMinute && leaveEndMin > startMinute`

**Search for tests:** `[VERIFIED: ls src/lib/search/__tests__/]` — `engine.test.ts` exists. `[VERIFIED: grep "leaveStart\|hasRecurringLeave\|isMultiDay"]` — **no tests reference these names.** Coverage gap: the multi-day-leave path is tested only by accident through tutor-level leave assertions.

**Real-world trigger analysis:** A leave from "Mon 14:00 to Tue 16:00" is `isMultiDay=true` → blocks entire Tuesday — correct. A leave from "Mon 14:00 to Mon 16:00" is `isMultiDay=false` → uses `leaveStart=Mon 14:00`, `leaveEnd=Mon 16:00` for minute math — correct, single weekday. **The bug as described in CONCERNS.md does not actually fire** because the only path where `current.getDay() === weekday` AND `isMultiDay === false` is when leaveStart and current are on the same day (single-day leave). The code is correct, just confusingly written.

**Recommendation:** **Option (b) — keep current behavior, document the assumption.** The "minute-of-day from day-cursor" alternative (option a) is conceptually neater but functionally equivalent for the single-day case and behaviorally identical to the existing short-circuit for the multi-day case.

**Recommended fix (option b):**

```typescript
function hasRecurringLeaveConflict(
  group: IndexedTutorGroup,
  weekday: number,
  startMinute: number,
  endMinute: number
): boolean {
  for (const leave of group.leaves) {
    const leaveStart = leave.startTime;
    const leaveEnd = leave.endTime;

    // Multi-day leaves block every day they touch — no minute-of-day math
    // because hours from leaveStart vs leaveEnd would lie about middle days.
    // Documented assumption: a leave spanning >24h is a full-day-block on
    // every day-of-week the range covers.
    const isMultiDay = leaveEnd.getTime() - leaveStart.getTime() > 24 * 60 * 60 * 1000;

    if (isMultiDay) {
      // Walk each day in [leaveStart, leaveEnd]; block if any day matches weekday.
      const cursor = new Date(leaveStart);
      while (cursor <= leaveEnd) {
        if (cursor.getDay() === weekday) return true;
        cursor.setDate(cursor.getDate() + 1);
      }
      continue;
    }

    // Single-day leave — leaveStart and leaveEnd are on the same calendar day.
    // Use leaveStart's weekday and minute-of-day directly (no day-cursor needed).
    if (leaveStart.getDay() !== weekday) continue;
    const leaveStartMin = leaveStart.getHours() * 60 + leaveStart.getMinutes();
    const leaveEndMin = leaveEnd.getHours() * 60 + leaveEnd.getMinutes();
    if (leaveStartMin < endMinute && leaveEndMin > startMinute) return true;
  }
  return false;
}
```

This refactor (a) eliminates the misleading variable use, (b) makes the multi-day vs single-day branch explicit, (c) is functionally equivalent to today's behavior. Existing tests continue to pass.

**Test scope:** Add 2 unit tests — (i) multi-day leave from Mon 14:00 to Wed 10:00 blocks Tue full-day slot; (ii) single-day leave from Mon 14:00 to Mon 16:00 does NOT block Tue 14:00 slot. Both tests will pass against the existing code AND the refactor — they exist to lock in the documented assumption.

**Confidence:** HIGH. Code reading + bug-trigger analysis confirms the existing behavior is correct but misleadingly named.

### REL-05 — Wise client retry policy

**Cited site:** `src/lib/wise/client.ts:67-91` (`fetchWithRetry`)

**Research question:** Right list of retryable codes? Does existing code classify?

**Answer:**
- Standard retryable codes per `[CITED: oneuptime.com/blog/post/2026-01-06-nodejs-retry-exponential-backoff/view]` and `[CITED: 1xapi.com/blog/resilient-api-circuit-breaker-bulkhead-retry-nodejs-2026]`: **5xx (500/502/503/504), 408 Request Timeout, 429 Too Many Requests, network errors (TypeError on fetch)**.
- Wise-specific codes: `[VERIFIED: src/lib/wise/types.ts]` defines no error type. The error path uses generic `throw new Error(\`Wise API ${response.status}: ${text}\`)` (client.ts:79).
- Existing code: classifies NOTHING. Lines 83-90 catch ANY error (network OR `!response.ok`) and retries up to 3 times. A 401, 404, 422 wastes ~7 seconds and 3 extra API calls.

**Recommended fix:**

```typescript
// client.ts:67-91 replacement

private static RETRYABLE_STATUS_CODES = new Set([408, 429, 500, 502, 503, 504]);

private async fetchWithRetry<T>(
  url: string,
  init: RequestInit,
  attempt = 0,
): Promise<T> {
  let response: Response;
  try {
    response = await fetch(url, {
      ...init,
      headers: {
        ...this.headers,
        ...(init.headers as Record<string, string> | undefined),
      },
    });
  } catch (networkErr) {
    // Network-level error (DNS, ECONNRESET, etc.) — retry
    if (attempt < this.maxRetries) {
      await new Promise((r) => setTimeout(r, Math.pow(2, attempt) * 1000));
      return this.fetchWithRetry<T>(url, init, attempt + 1);
    }
    throw networkErr;
  }

  if (response.ok) {
    return (await response.json()) as T;
  }

  const text = await response.text().catch(() => "");

  // Permanent error path: 4xx (except 429) — do NOT retry.
  if (!WiseClient.RETRYABLE_STATUS_CODES.has(response.status)) {
    throw new Error(`Wise API ${response.status}: ${text} (${url})`);
  }

  // Retryable error path: 5xx, 408, 429.
  if (attempt < this.maxRetries) {
    await new Promise((r) => setTimeout(r, Math.pow(2, attempt) * 1000));
    return this.fetchWithRetry<T>(url, init, attempt + 1);
  }
  throw new Error(`Wise API ${response.status}: ${text} (${url})`);
}
```

**Rejected enhancement (out of REL-05 scope, defer to v1.2 if needed):** Honor `Retry-After` header on 429/503. Adds value but expands scope; flag for OPS-02 / OPS-04 follow-up.

**Test scope:** 4 unit tests using `vi.fn()`-mocked fetch — (i) 401 throws on first try, no retry; (ii) 500 retries 3 times then throws; (iii) 429 retries; (iv) network error retries.

**Confidence:** HIGH. Standard pattern, well-documented.

### REL-07 — Timing-safe CRON_SECRET compare

**Cited site:** `src/app/api/internal/sync-wise/route.ts:11-17`

**Research question:** Correct Node.js pattern for `crypto.timingSafeEqual` against an HTTP header?

**Answer:** `[CITED: developers.cloudflare.com/workers/examples/protect-against-timing-attacks/]` and `[CITED: nodejs.org/api/crypto.html#cryptotimingsafeequala-b]` (per knowledge cutoff). The canonical pattern:

```typescript
import { timingSafeEqual } from "node:crypto";

function safeCompare(received: string, expected: string): boolean {
  const a = Buffer.from(received);
  const b = Buffer.from(expected);
  // Length-mismatch path: compare a against itself (always true) and negate.
  // This avoids both the RangeError throw AND leaking the secret length.
  if (a.length !== b.length) {
    return !timingSafeEqual(a, a);
  }
  return timingSafeEqual(a, b);
}
```

**Recommended fix:**

```typescript
// sync-wise/route.ts:10-17 replacement
import { timingSafeEqual } from "node:crypto";

async function handleSync(request: NextRequest) {
  const authHeader = request.headers.get("authorization") ?? "";
  const cronSecret = process.env.CRON_SECRET;
  if (!cronSecret) {
    return NextResponse.json({ error: "Server misconfigured" }, { status: 500 });
  }

  const expected = `Bearer ${cronSecret}`;
  const received = Buffer.from(authHeader);
  const known = Buffer.from(expected);

  // Constant-time compare; length-mismatch path avoids RangeError and leak.
  const valid =
    received.length === known.length &&
    timingSafeEqual(received, known);

  if (!valid) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
  // ... rest of handler
}
```

(The `length !== known.length && timingSafeEqual(...)` short-circuit returns `false` before throwing — and the short-circuit itself is constant-time relative to the secret since both branches do the same length comparison. The Cloudflare `!timingSafeEqual(a,a)` form is also valid; use whichever reads cleaner. **Recommended:** the short-circuit form because it's a single line and Node `timingSafeEqual` is documented to throw on mismatch — explicit length guard makes the intent clear.)

**Test scope:** 2 unit tests — (i) valid token returns 200; (ii) invalid token returns 401; (iii) wrong-length token returns 401 without throwing. Trivially testable via direct handler call with a `NextRequest` mock.

**Confidence:** HIGH. Pattern is well-documented; Node stdlib only.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Two-step UPDATE for atomic flip | Single-statement `SET active = (id = $newId)` with bounded WHERE | PG MVCC has had this since 7.x; just under-utilized | No production downtime window during promotion |
| `if (!cache) { await ... }` lazy init | Singleton-promise pattern (assign before await) | Pattern documented since 2020 (Mellman); pre-`AsyncMutex` libraries | Coalesces concurrent rebuilds without library cost |
| `===` on bearer tokens | `crypto.timingSafeEqual` with length guard | Pre-Node-stdlib (basic-auth #39, 2017); now stdlib-recommended | Removes theoretical timing-attack surface |
| Retry-everything | Status-code-aware retry (5xx + 408 + 429 + network) | Pre-2020 retry libs assumed all errors transient; modern resilience patterns gate by status | Faster permanent-error failures; less wasted budget |
| `new Date(toLocaleString(...))` for "now in TZ" | `toZonedTime(new Date(), TZ)` | `date-fns-tz` v1+ since 2019; project already uses it via `timezone.ts` | Removes Date-instant lie; future arithmetic stays correct |

## Open Questions

1. **`identity_collision` — new enum value or reuse `"alias"`?**
   - What we know: `dataIssueTypeEnum` does not include `identity_collision`. Migration via drizzle-kit costs ~1 plan.
   - What's unclear: Operator preference — distinct filter in `/data-health` UI vs. message-substring filter.
   - Recommendation: Reuse `"alias"` with `severity: "high"` and message prefix `identity_collision:`. Cheaper. If user wants distinct UI in 8.7 OPS-04 (data-health visibility), add a migration there.

2. **REL-04 keep-or-rewrite the day-cursor walk?**
   - What we know: The existing day-walk (`while (current <= leaveEnd)`) is functionally correct due to the multi-day short-circuit.
   - What's unclear: Whether refactoring for clarity (option b above) introduces any regression risk worth the LOC churn.
   - Recommendation: Refactor for clarity. The new shape is simpler and tests are cheap.

3. **REL-01 — bounded WHERE vs full-table UPDATE?**
   - Both are atomic; bounded WHERE creates fewer dead tuples (post-OPS-01 retention adds value but isn't critical now).
   - Recommendation: Bounded WHERE (`OR id = newId`). Cleaner intent.
