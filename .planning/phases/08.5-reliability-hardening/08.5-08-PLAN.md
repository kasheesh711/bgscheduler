Replace every occurrence of the `new Date(new Date().toLocaleString("en-US", { timeZone: "Asia/Bangkok" }))` idiom with `toZonedTime(new Date(), TIMEZONE)` from `date-fns-tz`. The 4 cited files contain a total of 6 occurrences (some files contain duplicates within the same component lifecycle). Purpose: Per CONCERNS.md and RESEARCH.md, the existing pattern produces a Date whose UTC fields appear to be Bangkok-local but is interpreted by all other Date code as UTC. Subsequent arithmetic with `.getDay()` / `.getHours()` accidentally works because the internal fields match — but the resulting Date is NOT a valid instant. Future code that uses `.getTime()` or compares instants will subtly break. Replacing with the canonical `toZonedTime` produces a properly-formed Date whose component-method outputs are identical for these read paths.
Output: 4 files updated to use `toZonedTime`; 1 lock-in test added to `timezone.test.ts`.
</objective>
