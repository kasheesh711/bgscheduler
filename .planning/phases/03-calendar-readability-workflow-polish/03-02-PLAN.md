**Step C — Local flash state + handler.**

Inside `AvailabilityGrid` (top of function body, before the empty-state guard):

```tsx
const [flashedId, setFlashedId] = useState<string | null>(null);

const handleQuickAdd = (e: React.MouseEvent, id: string, name: string) => {
  e.stopPropagation();
  if (disableAdd) return;
  onAddSingle(id, name);
  setFlashedId(id);
  window.setTimeout(() => {
    setFlashedId((current) => (current === id ? null : current));
  }, 800);
};
```

Use `window.setTimeout` explicitly (this is a browser client component — `"use client"` directive ensures no SSR execution).

## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | `compare.addTutor(id, name)` invocation from quick-add button | accept | `id` and `name` originate from the same API response the rest of the search UI trusts. The server-side boundary for compare (`/api/compare`) validates tutor IDs against the active snapshot independently. |
| T-03-05 | Denial of Service | Rapid "+" clicks spamming `addTutor` | mitigate | `addTutor` has an internal guard (`if (compareTutors.length >= 3) return`). We mirror it visually with `disableAdd` + HTML `disabled` attribute. Flash state does not block clicks but `addTutor`'s internal idempotency (adding same tutor twice is not possible because combobox/discovery already filter; quick-add relies on the compare state; duplicates would be a UX issue, not security). |
| T-03-06 | Information Disclosure | Native `title` tooltip on "+" button | accept | Tooltip content is static ("Add to compare" / "Remove a tutor first (max 3)"), no sensitive data. |

</threat_model>
