# Phase 08.6 — Test Coverage Hardening — Research

**Researched:** 2026-04-30
**Domain:** Test infrastructure (Vitest 4 + Drizzle/Postgres + Next.js 16 App Router + Auth.js v5)
**Confidence:** HIGH

## Summary

This phase is **test code only**. It adds ~60 tests across 7 plans to close HIGH-risk untested codepaths. The biggest open question — "how do we run integration tests offline against a real Postgres without breaking the locked stack?" — is answered by `@testcontainers/postgresql` v11.14.0 plus a parallel `drizzle-orm/node-postgres` driver path used **only in integration tests**, while production keeps `drizzle-orm/neon-http`. Both drivers consume the identical `drizzle/` migration directory via their respective programmatic `migrate()` functions, so there is no migration drift risk.

Three operational details drive the entire plan partition:

1. **The Neon HTTP driver does not support multi-statement transactions.** Test isolation must use `TRUNCATE … CASCADE` between tests, not transaction rollback. The orchestrator integration test (TCOV-02) and diff-hook integration test (TCOV-04) both share the same `truncateAll()` helper.
2. **Vitest 4 `projects` feature replaces the legacy workspace config.** Two projects (`unit`, `integration`) defined inline in `vitest.config.ts` give us `--project unit` (default fast) vs `--project integration` (testcontainers, longer timeout) without a second config file.
3. **Next.js 16 App Router route handlers are plain async functions** that take a `NextRequest` and return a `NextResponse`. They can be imported and called directly in Vitest with no Next runtime — but `auth()` must be `vi.mock()`-ed at the top of the test file (import order matters: mocks must be hoisted via `vi.mock()` before the route module is imported).

**Primary recommendation:** Add testcontainers + node-postgres + coverage-v8 as devDependencies. Adopt the `vitest.config.ts` `projects` pattern. For each TCOV-* requirement, use the verbatim code snippets and assertion lists in §"Per-Requirement Test Spec" below.

### Pattern 1: Vitest 4 `projects` config (REPLACES current `vitest.config.ts`)

**What:** Single config file defines two projects with different include patterns and timeouts.
**When to use:** When you want a fast `npm test` AND an opt-in `npm run test:integration`.
**Example:**
```ts
// vitest.config.ts (UPDATED — replaces current 14-line file)
// Source: https://vitest.dev/guide/projects (verified 2026-04-30)
import { defineConfig } from "vitest/config";
import path from "path";

export default defineConfig({
  resolve: {
    alias: {
      "@": path.resolve(__dirname, "./src"),
    },
  },
  test: {
    coverage: {
      provider: "v8",
      include: ["src/**/*.{ts,tsx}"],
      exclude: [
        "src/**/__tests__/**",
        "src/**/*.test.ts",
        "src/**/*.integration.test.ts",
        "src/tests/**",
        "src/app/**/*.tsx", // UI components — Phase 8.6 scope is server/data layer
      ],
      reporter: ["text", "html"],
    },
    projects: [
      {
        extends: true,
        test: {
          name: "unit",
          environment: "node",
          globals: true,
          include: ["src/**/*.test.ts", "src/**/*.test.tsx"],
          exclude: ["src/**/*.integration.test.ts"],
        },
      },
      {
        extends: true,
        test: {
          name: "integration",
          environment: "node",
          globals: true,
          include: ["src/**/*.integration.test.ts"],
          pool: "forks",
          poolOptions: { forks: { singleFork: true } },
          testTimeout: 60_000,
          hookTimeout: 60_000,
        },
      },
    ],
  },
});
```

The `extends: true` makes each project inherit the top-level `resolve.alias`. The integration project pins `pool: 'forks'` with `singleFork: true` so all integration tests share one process and one container — critical because spinning up a container per file would cost ~5s × N files.

### Pattern 2: Testcontainers + drizzle-orm/node-postgres bootstrap

**What:** A single helper that all integration tests import.
**When to use:** TCOV-02 and TCOV-04 (both need a real Postgres).
**Example:**
```ts
// src/tests/integration/db-helper.ts (NEW)
// Source: https://node.testcontainers.org/modules/postgresql/ (verified 2026-04-30)
// Source: https://orm.drizzle.team/docs/get-started-postgresql (verified 2026-04-30)
import { PostgreSqlContainer, StartedPostgreSqlContainer } from "@testcontainers/postgresql";
import { drizzle } from "drizzle-orm/node-postgres";
import { migrate } from "drizzle-orm/node-postgres/migrator";
import { sql } from "drizzle-orm";
import { Pool } from "pg";
import * as schema from "@/lib/db/schema";
import path from "path";

export type TestDb = ReturnType<typeof drizzle<typeof schema>>;

interface Handle {
  db: TestDb;
  pool: Pool;
  container: StartedPostgreSqlContainer;
}

export async function startTestDb(): Promise<Handle> {
  const container = await new PostgreSqlContainer("postgres:16-alpine")
    .withDatabase("bgscheduler_test")
    .withUsername("test")
    .withPassword("test")
    .start();

  const pool = new Pool({ connectionString: container.getConnectionUri() });
  const db = drizzle(pool, { schema });

  await migrate(db, {
    migrationsFolder: path.resolve(__dirname, "../../../drizzle"),
  });

  return { db, pool, container };
}

export async function stopTestDb(h: Handle): Promise<void> {
  await h.pool.end();
  await h.container.stop();
}

/**
 * Truncate every data table between tests. Schema (enums, indexes) preserved.
 * Order does not matter because CASCADE follows FK chains.
 */
export async function truncateAll(db: TestDb): Promise<void> {
  // Listed by name (snake_case) to match the actual Postgres tables, not the
  // camelCase TypeScript identifiers. Safe with CASCADE — the DB sorts FK order.
  await db.execute(sql`
    TRUNCATE TABLE
      data_issues,
      snapshot_stats,
      future_session_blocks,
      past_session_blocks,
      dated_leaves,
      recurring_availability_windows,
      raw_teacher_tags,
      subject_level_qualifications,
      tutors,
      tutor_identity_group_members,
      tutor_identity_groups,
      tutor_aliases,
      admin_users,
      sync_runs,
      snapshots
    RESTART IDENTITY CASCADE
  `);
}
```

**Driver swap rationale:** `@neondatabase/serverless` (the prod driver) speaks Neon's HTTP API and **cannot connect to a generic Postgres TCP port**. The `drizzle-orm/node-postgres` driver accepts a `pg.Pool` and works against any Postgres ≥9.5 — including the testcontainer. The Drizzle query API (`db.select().from(...)`, `db.insert(...).values(...)`) is **identical across drivers** because schema is the source of truth. Our production code under test (`runFullSync`, `ensureIndex`, etc.) takes a `Database` parameter typed as `ReturnType<typeof getDb>` — TypeScript will accept the test handle because both drivers' `drizzle(...)` returns the same structural type when given the same schema.

**What:** Direct import + manually-built `NextRequest`.
**When to use:** All 8 routes in TCOV-03.

Mirror the `makeTutor()` / `makeIndex()` helpers from `src/lib/search/__tests__/engine.test.ts:6-42`. ### Anti-patterns to avoid

- **Spying on internal function calls via `vi.spyOn(module, 'fn')`** — ESM live bindings prevent this from intercepting same-module calls. The 08.5-02 plan documented this trap and used DB-call counting instead. Stick to that pattern for any test that wants to verify "buildIndex was called once" — count `db.select` invocations on the fake DB, not the function.
- **Using `vi.mock()` without `factory`** — auto-mocking in Vitest 4 is brittle for Auth.js. Always pass `() => ({ ... })` factory.
- **Forgetting to await `request.json()` or `response.json()` in tests** — both are async; tests that don't await them silently pass with the unresolved promise.
- **Listing tables in TRUNCATE without RESTART IDENTITY** — leaves UUID sequences intact (low risk for our gen_random_uuid columns) but is also a footgun for any future SERIAL columns. Always use `RESTART IDENTITY CASCADE`.
- **Putting integration tests in the default include glob** — slows `npm test` from <1s to ~15s. The phase explicitly carves out `*.integration.test.ts` to keep the fast path fast.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Spin up a Postgres for tests | Bash script + `pg_ctl` + cleanup hooks | `@testcontainers/postgresql` | Container is sandboxed, auto-cleaned, version-pinned, parallel-safe |
| Apply Drizzle migrations to test DB | Read SQL files + execute one-by-one | `import { migrate } from "drizzle-orm/node-postgres/migrator"` | Tracks `drizzle/meta/_journal.json` ordering; identical to prod migrate path |
| Mock Auth.js v5 | Hand-build session object factory + spy on `getDb` | `vi.mock('@/lib/auth', () => ({ auth: vi.fn() }))` | Vitest's hoisted mock keeps tests fully synchronous and type-safe via `vi.mocked(auth).mockResolvedValue(...)` |
| Build a mock `NextRequest` | Plain `Request` shim with manual headers | `import { NextRequest } from "next/server"` | NextRequest extends Request and adds `nextUrl`, `cookies` — handlers reach for these |
| Coverage report | Custom byte-counter | `@vitest/coverage-v8` | Default provider in Vitest 4; one flag (`--coverage`) and it works |
| Truncate-cascade SQL | Per-test schema recreate | `db.execute(sql\`TRUNCATE … CASCADE\`)` | Schema recreate is 5+ seconds; truncate is <50ms |

**Key insight:** Every TCOV-* requirement has a 1:1 standard tool. The only custom code we write is the 80-line `db-helper.ts` that glues testcontainers + node-postgres + the existing schema together. Everything else is direct invocation of locked-stack libraries.

## Common Pitfalls

### Pitfall 1: Existing TS errors in `past-sessions-diff-hook.test.ts:185-186`
**What goes wrong:** The current test file's `PriorBlock` type (lines 15-33) lists fields from `futureSessionBlocks` (the source table), but assertions at lines 185-186 reference `groupCanonicalKey` and `capturedInSnapshotId` — fields that exist on `pastSessionBlocks` (the destination table), not the prior block. The author conflated source rows with inserted rows.
**Why it happens:** The diff-hook reads from `futureSessionBlocks` and writes to `pastSessionBlocks`; the two have different schemas (verified at `src/lib/db/schema.ts:222-258` and `drizzle/0002_past_session_blocks.sql`).
**How to avoid:** When TCOV-04 rewrites this file as `*.integration.test.ts`, **delete the local `PriorBlock` interface and `makePriorBlock` helper entirely**. Use Drizzle's inferred `typeof schema.futureSessionBlocks.$inferInsert` for source rows and `$inferSelect` for assertions on `past_session_blocks`. The integration test does not need a hand-rolled mock DB; it inserts directly via `db.insert(schema.futureSessionBlocks).values(...)`.
**Warning signs:** `tsc --noEmit` flags any access to `.groupCanonicalKey` on a `futureSessionBlocks` row.

### Pitfall 2: Neon HTTP driver in tests will throw on connect
**What goes wrong:** If a test imports `getDb()` (production code path) without first mocking `@/lib/db`, the Neon HTTP driver tries to `fetch()` against a non-existent endpoint and fails with a confusing error.
**Why it happens:** `getDb()` reads `process.env.DATABASE_URL` and constructs a Neon HTTP client at import time.
**How to avoid:** Two options depending on test type:
- **Unit/route tests:** `vi.mock('@/lib/db', () => ({ getDb: vi.fn() }))` and inject a fake or mocked DB. Already shown in Pattern 3.
- **Integration tests:** Bypass `getDb()` entirely. The orchestrator + diff-hook both accept a `Database` parameter via dependency injection — pass the test handle directly: `await runFullSync(testHandle.db, mockClient, instituteId)`.
**Warning signs:** Test fails with "fetch is not defined" or "ENOTFOUND" before any assertion runs.

### Pitfall 3: Vitest module mocks must come BEFORE the import they mock
**What goes wrong:** `vi.mock('@/lib/auth', ...)` placed after `import { POST } from '@/app/api/compare/route'` does not mock `auth()` for the route module — the route already captured the real reference.
**Why it happens:** Vitest hoists `vi.mock(...)` calls to the top of the file at parse time, but only `vi.mock` itself, not surrounding code. Imports run before any non-mock statements regardless of source order.
**How to avoid:** Always put `vi.mock(...)` calls at the top of the file, immediately after the type imports from vitest. The pattern is shown in §Pattern 3.
**Warning signs:** `vi.mocked(auth).mockResolvedValue(...)` throws "auth is not a mock function" — meaning the real `auth` was imported.

### Pitfall 4: `db.execute(sql\`TRUNCATE ...\`)` on neon-http vs node-postgres
**What goes wrong:** Both drivers expose `db.execute(sql)`, but the return type and behavior differ subtly. neon-http returns `{ rows, rowCount, ... }`; node-postgres returns the same shape. For TRUNCATE no rows are returned, so neither matters — but blindly using one driver's snippet against the other can mislead. The `truncateAll()` helper above uses node-postgres-only `db`, so this is contained, but **do not accidentally use the production neon-http driver in the helper**.
**How to avoid:** The helper imports `drizzle` from `drizzle-orm/node-postgres`. Production code imports from `drizzle-orm/neon-http`. If both imports appear in the same test file, that test is doing something wrong.
**Warning signs:** The integration test compiles but throws at runtime with "this.client is not a function" — the wrong driver was used to wrap the pg pool.

### Pitfall 5: Integration test container leaks if `afterAll` is skipped
**What goes wrong:** A failed `beforeAll` (e.g., Docker not running) doesn't trigger `afterAll`, so the partially-started container lingers. Subsequent runs accumulate.
**How to avoid:** Wrap container start in a try/catch that calls `container.stop()` on error. Even better: use the `await using` pattern (TC39 explicit resource management, supported by testcontainers v11) so cleanup is automatic.
**Warning signs:** `docker ps` shows orphan `postgres:16-alpine` containers between test runs.

### Pitfall 6: `vi.resetAllMocks()` vs `vi.restoreAllMocks()` confusion
**What goes wrong:** `resetAllMocks` clears mock state but keeps mock implementations. `restoreAllMocks` reverts to original implementations. The 08.5-02 test uses both. For TCOV-03, `resetAllMocks()` in `beforeEach` is the right choice because `vi.mock()` factories at module top set the mock identity; we only want to reset the per-test state.
**How to avoid:** `beforeEach(() => vi.resetAllMocks())` + re-establishing default mock returns after the reset. Don't mix.

### Pitfall 7: Coverage exclusions hide real gaps
**What goes wrong:** Default Vitest 4 coverage no longer reports uncovered files unless they're in `coverage.include` (this was a v3→v4 breaking change per Vitest docs). Forgetting to set `coverage.include` produces optimistic-looking percentages.
**How to avoid:** The `vitest.config.ts` snippet in Pattern 1 explicitly sets `include: ['src/**/*.{ts,tsx}']`. If a future plan disables this, the verifier should catch it.
**Warning signs:** Coverage report shows 0 files when there should be 50+.

### Pitfall 8: Sync orchestrator integration test hangs on Wise client
**What goes wrong:** `runFullSync(db, client, instituteId)` calls `fetchAllTeachers(client, ...)` which makes real HTTP calls to wiseapp.live. In a test, this either times out or pollutes prod data.
**How to avoid:** Pass a fake `WiseClient`. The integration test for TCOV-02 is about DB persistence and promotion logic, not Wise fetching. Stub the client interface (request/response is well-typed in `src/lib/wise/types.ts`) and inject. See §"Per-Requirement Test Spec" TCOV-02 for the exact stub shape.
**Warning signs:** Integration test runs longer than 10s — likely hitting real Wise API.

beforeAll(async () => { handle = await startTestDb(); }, 60_000);
afterAll(async () => { await stopTestDb(handle); });

async function seedPriorSnapshot(db: typeof handle.db, opts: {
  groupCanonicalKey: string;
  pastWiseSessionIds: string[];
  futureWiseSessionIds: string[];
}) {
  const [snap] = await db.insert(schema.snapshots).values({ active: true })
    .returning({ id: schema.snapshots.id });

  const [group] = await db.insert(schema.tutorIdentityGroups).values({
    snapshotId: snap.id,
    canonicalKey: opts.groupCanonicalKey,
    displayName: "Test",
    supportedModality: "onsite",
  }).returning({ id: schema.tutorIdentityGroups.id });

  const PAST = new Date(Date.now() - 24 * 3600_000);
  const FUTURE = new Date(Date.now() + 24 * 3600_000);

  const rows = [
    ...opts.pastWiseSessionIds.map((id) => ({
      snapshotId: snap.id, groupId: group.id, wiseTeacherId: "t1",
      wiseSessionId: id, startTime: PAST, endTime: new Date(PAST.getTime() + 3600_000),
      weekday: PAST.getDay(), startMinute: 600, endMinute: 660,
      wiseStatus: "CONFIRMED", isBlocking: true,
    })),
    ...opts.futureWiseSessionIds.map((id) => ({
      snapshotId: snap.id, groupId: group.id, wiseTeacherId: "t1",
      wiseSessionId: id, startTime: FUTURE, endTime: new Date(FUTURE.getTime() + 3600_000),
      weekday: FUTURE.getDay(), startMinute: 600, endMinute: 660,
      wiseStatus: "CONFIRMED", isBlocking: true,
    })),
  ];
  if (rows.length) await db.insert(schema.futureSessionBlocks).values(rows);

  return { snapshotId: snap.id, groupId: group.id };
}

describe("runPastSessionsDiffHook — integration (real Postgres)", () => {
  it("captures dropped past sessions into past_session_blocks", async () => {
    await seedPriorSnapshot(handle.db, {
      groupCanonicalKey: "tutor-a",
      pastWiseSessionIds: ["S-past-1", "S-past-2"],
      futureWiseSessionIds: ["S-future-1"],
    });

    // New snapshot row (not yet active)
    const [newSnap] = await handle.db.insert(schema.snapshots).values({ active: false })
      .returning({ id: schema.snapshots.id });

    // New Wise response only contains S-future-1 — past ones dropped
    const newSessions: NormalizedSessionBlock[] = [{
      wiseSessionId: "S-future-1", wiseTeacherId: "t1",
      startTime: new Date(Date.now() + 24 * 3600_000),
      endTime: new Date(Date.now() + 25 * 3600_000),
      weekday: 0, startMinute: 600, endMinute: 660,
      wiseStatus: "CONFIRMED", isBlocking: true,
    }];

    const result = await runPastSessionsDiffHook(handle.db as never, newSessions, newSnap.id);

    expect(result.capturedCount).toBe(2);

    const captured = await handle.db.select().from(schema.pastSessionBlocks);
    expect(captured.map((r) => r.wiseSessionId).sort()).toEqual(["S-past-1", "S-past-2"]);
    expect(captured.every((r) => r.groupCanonicalKey === "tutor-a")).toBe(true);
    expect(captured.every((r) => r.capturedInSnapshotId === newSnap.id)).toBe(true);
  });

  it("idempotent: second run doesn't double-insert (UNIQUE constraint)", async () => {
    await seedPriorSnapshot(handle.db, {
      groupCanonicalKey: "tutor-a",
      pastWiseSessionIds: ["S1"],
      futureWiseSessionIds: [],
    });
    const [newSnap] = await handle.db.insert(schema.snapshots).values({ active: false })

    expect(r1.capturedCount).toBe(1);

    const r2 = await runPastSessionsDiffHook(handle.db as never, [], newSnap.id);
    expect(r2.capturedCount).toBe(0); // ON CONFLICT DO NOTHING

    const allCaptured = await handle.db.select().from(schema.pastSessionBlocks)
      .where(eq(schema.pastSessionBlocks.wiseSessionId, "S1"));
    expect(allCaptured).toHaveLength(1);
  });
});

## Open Questions (RESOLVED)

1. **Should TCOV-03 cover all 4 status codes (200/401/400/500) on every route, or scale?**
   - What we know: CONTEXT Area 2 says "auth (200/401), validation (400 with Zod errors), success (200 with shape), 500 error path". That's 4 cases per route × 8 routes = 32 tests minimum.
   - What's unclear: Some routes (e.g., `tutors`, `filters`) have no validation step (no body) — Zod 400 doesn't apply. Those routes should have 3 cases each.
   - Recommendation: Planner counts ~28-32 tests across the 8 routes. Expect a partition like 4 cases × 6 routes (compare, compare/discover, search, search/range, data-health body-less, sync-wise) + 3 cases × 2 routes (tutors, filters).

2. **Does the orchestrator integration test for "fail-mid-promotion" need to simulate a network failure mid-UPDATE, or is the >50% unresolved-ratio path sufficient?**
   - What we know: REL-01 already collapsed the two-statement promotion into one atomic UPDATE. The window where zero rows match `active = true` no longer exists.
   - What's unclear: TCOV-02's "fail-mid-promotion" might mean (a) test the `unresolvedRatio >= 0.5 → shouldPromote=false` path, OR (b) test the outer try/catch in the orchestrator.
   - Recommendation: Test (a) only. Path (b) requires injecting a DB error mid-statement which is impossible without a chaos-monkey driver wrapper. Document (b) as outside the scope of integration testing; the catch handler is already covered by REL-06 unit logic.

3. **Should `truncateAll()` include `admin_users` and `tutor_aliases` (which are seeded data, not snapshot-scoped)?**
   - What we know: `admin_users` is read by the auth signIn callback; `tutor_aliases` is read by the orchestrator. Both are seeded once and rarely change.
   - What's unclear: Truncating them in `beforeEach` means each integration test must re-seed before invoking code that reads them.
   - Recommendation: Yes, include them. Each integration test that needs admins or aliases should explicitly seed them in its setup. This makes test intent explicit and prevents test ordering dependencies.
