## Gray areas — auto-selected defaults

Each area below was auto-selected per Auto-mode. Rationale + alternatives included so the user can reopen if needed.

### Area 1 — Integration-test database strategy (TCOV-02, TCOV-04)
**AUTO-LOCKED:** Local Docker Postgres via `@testcontainers/postgresql`.

- **Why:** Reproducible offline, no Neon test-branch budget needed (Hobby plan), spins up a per-suite Postgres 16 container with the Drizzle migrations applied, tears down after. Already a pattern Drizzle docs recommend.
- **Cost:** Adds 2 dev dependencies (`@testcontainers/postgresql`, `testcontainers`). Containers start in <5s on a warm Docker daemon — not run in CI by default (separate `npm run test:integration` script).
- **Alternatives considered:**
  - Neon test branch — operational cost + Neon API plumbing; rejected for Hobby budget.
  - In-memory SQLite — impossible: schema uses pgEnum and Postgres-specific JSONB constructs.
  - PGlite (browser Postgres) — promising but immature for the Drizzle Neon driver path; revisit in 8.7.

### Area 2 — API route test invocation (TCOV-03)
**AUTO-LOCKED:** Direct handler import + manually-constructed `Request` object.

- **Why:** Next.js 16 App Router route files export named handlers (`GET`, `POST`). Importing them and calling with a `new Request(url, { method, headers, body })` is the lowest-overhead approach, matches the existing minimal-mock test style, no extra dependency.
- **Pattern:**
  ```ts
  import { POST } from "@/app/api/compare/route";
  const res = await POST(new NextRequest("http://localhost/api/compare", { method: "POST", body: JSON.stringify(...) }));
  expect(res.status).toBe(200);
  ```
- **Alternatives considered:**
  - `next-test-api-route-handler` — abandons the Next.js 16 App Router pattern, mostly designed for Pages Router.
  - supertest against `next dev` — slow, flaky, requires server lifecycle in test setup.

### Area 3 — Auth mocking strategy (TCOV-03 protected routes, TCOV-06)
**AUTO-LOCKED:** `vi.mock('@/lib/auth', () => ({ auth: vi.fn() }))` per-test override.

- **Why:** Auth.js v5 `auth()` is the single gate. Mocking it gives per-test control over session shape (admin email vs non-admin vs missing). No real OAuth flow needed.
- **Pattern:**
  ```ts
  import { auth } from "@/lib/auth";
  vi.mock("@/lib/auth", () => ({ auth: vi.fn() }));
  beforeEach(() => vi.mocked(auth).mockResolvedValue({ user: { email: "kevhsh7@gmail.com" } } as any));
  ```
- **For TCOV-06 specifically**: also test the real `signIn` callback against a stubbed `db.select` returning the admin allowlist. That's where the allowlist logic lives — mocking `auth()` doesn't cover it.
- **Alternatives considered:**
  - Real OAuth — impractical in unit tests; not the auth surface being tested.
  - Stub the whole NextAuth handler — too coarse; loses the signIn-callback coverage TCOV-06 demands.

### Area 4 — Test isolation for integration suites (TCOV-02, TCOV-04)
**AUTO-LOCKED:** `TRUNCATE … CASCADE` of all data tables per test.

- **Why:** Drizzle migrations are heavy (Postgres extensions, enums, foreign keys); recreating per test is wasteful. Truncate-cascade is fast (<50ms), preserves schema, gives every test a clean slate. Avoids transaction-pinning quirks of the `@neondatabase/serverless` HTTP driver.
- **Pattern:** `beforeEach(async () => { await db.execute(sql\`TRUNCATE snapshots, sync_runs, ... CASCADE\`); })` — list maintained centrally in a `truncateAll()` helper.
- **Alternatives considered:**
  - Per-test transaction rollback — neon-http does not support cross-statement transactions; would need switch to `pg` driver for tests.
  - Recreate schema per test — too slow (5+ seconds per test).

### Area 5 — Coverage threshold CI gate (cross-cutting)
**AUTO-LOCKED:** Track via `vitest --coverage` locally, **no CI gate yet**. Defer to Phase 8.7.

- **Why:** Bootstrap coverage in this phase first. Adding a CI threshold before the baseline exists creates a chicken-and-egg flake risk. Once the floor is established at the end of 8.6, Phase 8.7 OPS work can ratchet it into CI with a confident initial threshold.
- **What this phase does:** Run `npm run test:coverage` after each plan's tests are added; document the percentage delta in each SUMMARY.md so the verifier can see the trend.
- **Alternative considered:** Hard 70% line / 60% branch threshold immediately — rejected; the codebase has untested code paths (UI components, error branches in non-critical paths) that would force a wider scope.

### Area 6 — `globalThis` singleton hygiene (TCOV-01)
**AUTO-LOCKED:** Reset `globalThis.__bgscheduler_*` keys in `beforeEach` of any test that touches the search index. Pattern already established in 8.5-02's reproducer.

- **Why:** Vitest runs files in worker pools but tests within a file share globals. The singleton-promise pattern in `src/lib/search/index.ts` keys off `globalThis.__bgscheduler_searchIndex` and `__bgscheduler_searchIndexBuildPromise`. Without reset, snapshot-mismatch tests interfere with each other.
- **Pattern:**
  ```ts
  beforeEach(() => {
    delete (globalThis as any).__bgscheduler_searchIndex;
    delete (globalThis as any).__bgscheduler_searchIndexBuildPromise;
  });
  ```

### Area 7 — Test file naming + location (TCOV-02, TCOV-04)
**AUTO-LOCKED:** Co-located `{module}.integration.test.ts` in existing `__tests__/` directories.

- **Why:** Matches the existing co-location convention. Filter via filename in the run script — `vitest run **/*.test.ts` for unit, `vitest run **/*.integration.test.ts` for integration. Adds two `package.json` scripts: `test:unit` and `test:integration`. Default `npm test` continues to run only unit tests (keeps `npm test` <1s).
- **Alternative considered:** Separate `tests/integration/` directory — breaks co-location convention, makes it harder to find the test for a given module.

### Area 8 — Modality contradiction test data setup (TCOV-07)
**AUTO-LOCKED:** Stub `db.insert` for `data_issues` table; assert it was called with `type: 'conflict_model'` payload when `detectSessionModalityConflict` fires.

- **Why:** TCOV-07 verifies the **orchestrator's persistence path**, not the conflict-detection logic itself (already covered in `modality.test.ts`). Stubbing the DB layer isolates that path without requiring integration-test infrastructure. Faster, more focused, no DB dependency.
- **Pattern:**
  ```ts
  const insertSpy = vi.fn();
  vi.mocked(db.insert).mockReturnValue({ values: insertSpy } as any);
  // ... trigger orchestrator with conflicting-modality fixture
  expect(insertSpy).toHaveBeenCalledWith(expect.objectContaining({ type: 'conflict_model' }));
  ```
- **Alternative considered:** Bundle into TCOV-02 integration suite — possible, but adds suite complexity for a check that doesn't need real DB.
