# Phase 5 — UI Design Contract

> Visual and interaction contract for the POLISH Drain phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Phase posture:** This is a polish phase, not a feature phase. The ONLY shape-changing UI work is POLISH-09 (swap hardcoded `bg-red-500` → semantic `--today-indicator` token). Everything else is human QA against production UI that already shipped in v1.0 / v1.0.1. This contract inherits the v1.0 design system unchanged, locks the new `--today-indicator` token exactly once, and specifies the human-QA walkthrough checklist so the executor can record sign-off without guessing.

### NEW in Phase 5 — `--today-indicator` (POLISH-09)

| Property | Value |
|----------|-------|
| CSS custom property | `--today-indicator` |
| `@theme inline` alias | `--color-today-indicator: var(--today-indicator)` |
| Light-mode value | `oklch(0.628 0.2577 29.23)` |
| Dark-mode value | `oklch(0.628 0.2577 29.23)` (identical — GCal convention) |
| Visual equivalent | `#ef4444` (Tailwind `red-500`) — zero visual regression |
| Tailwind utility | `bg-today-indicator` |
| Semantic role | Current-moment indicator ("now line" + dot) in weekly calendar views |
| Usage rule | ONLY at the four call sites below. NOT for badges, buttons, errors, destructive actions, or any other UI surface. |
| Rationale | Preserves GCal/Outlook/Cron convention (always-red "now" line); decoupled from `--destructive` because "current moment" is not an error/danger signal (CONTEXT.md D-06 rejects `--destructive` reuse). |

**Call sites (exact line-level replacement, no other logic changes):**

| File | Line | Before | After |
|------|------|--------|-------|
| `src/components/compare/calendar-grid.tsx` | 303 | `className="absolute left-0 right-0 bg-red-500 z-[3] pointer-events-none"` | `className="absolute left-0 right-0 bg-today-indicator z-[3] pointer-events-none"` |
| `src/components/compare/calendar-grid.tsx` | 307 | `className="absolute h-2 w-2 rounded-full bg-red-500 z-[3] pointer-events-none"` | `className="absolute h-2 w-2 rounded-full bg-today-indicator z-[3] pointer-events-none"` |
| `src/components/compare/week-overview.tsx` | 547 | `className="absolute left-0 right-0 bg-red-500 z-[3] pointer-events-none"` | `className="absolute left-0 right-0 bg-today-indicator z-[3] pointer-events-none"` |
| `src/components/compare/week-overview.tsx` | 551 | `className="absolute h-2 w-2 rounded-full bg-red-500 z-[3] pointer-events-none"` | `className="absolute h-2 w-2 rounded-full bg-today-indicator z-[3] pointer-events-none"` |

**`globals.css` insertion points:**

```css
@theme inline {
  /* ...existing tokens unchanged... */
  --color-free-slot: var(--free-slot);
  --color-today-indicator: var(--today-indicator);   /* NEW */
  /* ...rest unchanged... */
}

:root {
  /* ...existing tokens unchanged... */
  --free-slot: oklch(0.72 0.17 155);
  --today-indicator: oklch(0.628 0.2577 29.23);      /* NEW — GCal red */
}

.dark {
  /* ...existing tokens unchanged... */
  --free-slot: oklch(0.72 0.17 155);
```

### POLISH-02 — Discovery modal error state

**Environment:**
- Chrome or Safari (either works; Chrome DevTools used for forced failure)
- Network tab open

**Reproduction:**

1. Navigate to `/search`
2. Click "Advanced search" in the right compare panel → Discovery modal opens
3. Enter any filters (subject + level) and click Search inside the modal
4. Force failure using ONE of:
   - **Method A (preferred, deterministic):** Chrome DevTools → Network tab → Block request URL pattern for `/api/compare/discover`
   - **Method B:** Disconnect Wi-Fi for 3 seconds during the request, reconnect after the request fires

**Pass criteria:**

| # | Criterion |
|---|-----------|
| 1 | Error surfaces visibly — toast, inline message, or modal banner (NOT silent) within 3s of the failed response |
| 2 | Error message uses semantic color (`--destructive` or `--conflict` — NOT literal hex) — verifiable via DevTools Elements panel computed styles |
| 3 | Error message is legible at default zoom (color contrast ≥ 4.5:1 against its background — approximate eyeball check) |
| 4 | User can dismiss or retry without refreshing the page (dismiss X on toast / retry button / modal close + re-open) |

**Fail handling:** If error is silent (criterion 1 fails), record the observed behavior (spinner hangs, modal resets, etc.) in `05-VERIFICATION.md`. Remediation rolls forward.

## Accessibility Expectations (POLISH-01 reference — WCAG 2.1 AA)

Baseline expectations for the VoiceOver walkthrough. These are the semantic-behavior contracts the executor should confirm, not new implementation asks.

| Category | Expectation | WCAG reference |
|----------|-------------|----------------|
| Keyboard navigation | All interactive elements reachable via Tab; visible focus ring on every focusable element | 2.1.1, 2.4.7 |

| Motion | No motion requirement introduced in this phase; existing `prefers-reduced-motion` paths respected if present | 2.3.3 |

**Known non-compliance accepted into this phase:** None. If the walkthrough surfaces a WCAG failure, it is recorded in `05-VERIFICATION.md` and escalated — it does NOT block Phase 5 sign-off unless it invalidates a stated POLISH-01 pass criterion.
