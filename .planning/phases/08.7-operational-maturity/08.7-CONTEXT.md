# Phase 8.7: Operational Maturity - Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Add the operational controls needed to run BGScheduler reliably for the next 12+ months: snapshot retention, in-app sync failure visibility, stale-data warning on the main workspace, manual admin sync from `/data-health`, dependency cleanup, exact version pinning, and the small low-risk code hygiene items already listed in OPS-07.

**In scope:** OPS-01..07. Snapshot pruning keeps storage bounded. Failed syncs are visible in-app. `/search` and `/compare` show a stale-data banner only when the latest successful sync is more than 48 hours old. `/data-health` gets a confirmed manual "Sync now" flow using authenticated admin session auth. Dependency cleanup removes unused runtime packages and pins riskier packages exactly.

**Out of scope:** External notification providers, new third-party services, Pro-plan cron changes, mobile redesign, sync-pipeline overhaul beyond pruning/manual trigger wiring, and any pruning of `past_session_blocks`.

</domain>

<decisions>
## Implementation Decisions

### Sync Failure Visibility

- **D-01:** Failed sync visibility is **in-app only**. `/data-health` is the primary failure details surface; the stale-snapshot banner and manual sync feedback are secondary surfaces. No external notification provider is implemented in Phase 8.7, and no notification-related env var is added.
- **D-02:** OPS-02 has been amended from external notification integration to in-app operational visibility. Downstream agents must follow the updated wording in `.planning/REQUIREMENTS.md` and `.planning/ROADMAP.md`, not the older audit recommendation.
- **D-03:** `/data-health` remains the canonical operational page for sync status: last successful sync, last failed sync, failure error, active snapshot, issue counts, and recent sync history.

### Stale Snapshot Banner

- **D-04:** The stale-data banner lives at the app level under the shared navigation, implemented through the authenticated `(app)` layout or a child component it renders. It appears consistently on `/search` and `/compare`.
- **D-05:** Banner trigger is **only** `staleAgeMs > 48h`. A latest failed sync does not show the banner if the active data is still fresh.
- **D-06:** Dismissal is per browser session via `sessionStorage`. A dismissed banner returns in a new browser session.
- **D-07:** Copy is plain admin warning, not urgent/destructive: `Tutor data may be outdated. Last successful sync was over 48 hours ago.` Include a `View data health` link to `/data-health`.
- **D-08:** The existing 26h stale threshold change in OPS-04 is separate from the 48h banner trigger. 26h controls API `snapshotMeta.stale` warnings so daily Hobby cron is not noisy; 48h controls the app-level operator banner.

### Manual "Sync Now" UX

- **D-09:** The manual sync button lives in the top sync status area of `/data-health`, near `Last Successful Sync` / `Last Failed Sync`.
- **D-10:** Clicking `Sync now` requires confirmation before starting. The confirmation should mention that sync can take several minutes and contacts Wise heavily.
- **D-11:** While sync is running, the button is disabled and shows a running/progress state. Duplicate clicks are prevented.
- **D-12:** After completion, the UI shows success/failure feedback inline or via an existing lightweight feedback pattern, then refreshes health data so cards and recent sync history update.
- **D-13:** `POST /api/internal/sync-wise` gets an authenticated admin-session path for browser manual sync. Vercel cron remains protected by `CRON_SECRET`; the browser never receives or sends the cron secret.
- **D-14:** Manual sync failure does not need a separate route. The same route can branch by auth mechanism: cron secret for scheduled calls; Auth.js session for authenticated admin POST calls.

### Snapshot Pruning Semantics

- **D-15:** Pruning keeps the latest 30 snapshots by `createdAt` and always preserves the active snapshot, even if timestamp ordering is unusual.
- **D-16:** Deletion is implemented with manual ordered deletes of snapshot-scoped tables: child tables first, then `snapshots`. Do not add broad `ON DELETE CASCADE` migration changes in this phase.
- **D-17:** Pruning runs after successful promotion and after the sync run has been updated as successful. The live data path completes before cleanup.
- **D-18:** If pruning fails after a successful sync, log the pruning failure and record metadata, but do not flip the sync run to failed. The new snapshot was promoted successfully; cleanup failure should be visible but not misrepresent data freshness.
- **D-19:** `past_session_blocks` is explicitly untouched by Phase 8.7 pruning. It is cross-snapshot by design and Phase 7 locked unbounded retention for v1.1.

### the agent's Discretion

- **Dependency cleanup:** OPS-06 is straightforward. Remove `@auth/drizzle-adapter`; move `shadcn` to `devDependencies`; document package/bundle impact in the phase summary or PR notes.
- **Version pinning:** OPS-07 package pins are mechanical: remove caret ranges from `next-auth`, `@base-ui/react`, and `drizzle-orm`. Let lockfile changes reflect the exact installed versions.
- **`detectConflicts` cleanup:** Remove the unused `indexedGroups` parameter and update call sites/tests.
- **Medium modality tier TODO:** Either implement a deliberate `confidence === "medium"` branch or replace the TODO with an explicit note that no code emits `medium` yet. The behavior must be intentional, not an open TODO.
- **Pruning metadata shape:** Planner chooses exact metadata keys, but they should capture enough to diagnose pruning: attempted count, deleted snapshot count, deleted row counts if practical, and error message on failure.
- **Banner implementation shape:** Planner chooses whether the stale status is fetched by a small client component in `AppLayout` or passed through a server helper, as long as it works on `/search` and `/compare`, is dismissible per session, and links to `/data-health`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- **`/data-health` data contract** already returns `lastSuccessfulSync`, `lastFailedSync`, `lastFailureError`, `staleAgeMs`, `staleMinutes`, active snapshot, issue counts, and recent syncs. Manual sync and banner work should reuse this shape or extend it minimally.
- **`AppLayout` route group** wraps `/search`, `/compare`, and `/data-health` with `AppNav`; it is the right integration point for a shared app-level banner.
- **`syncRuns.metadata` JSONB** already stores Phase 7 diff-hook metadata. Reuse this for pruning outcome metadata rather than adding a table.
- **`auth()`** is already the app-wide authenticated-session primitive. Manual sync POST should use this, not expose cron credentials.

### Established Patterns

- **API routes self-authenticate.** `/api/internal/*` bypasses middleware, so `sync-wise` must explicitly validate cron secret or admin session.
- **Fail-closed product rule still applies.** Operational UI must warn when stale, but search/compare availability rules remain unchanged.
- **No new stack pieces.** Operational maturity is built with existing Next.js route handlers, Auth.js session auth, Drizzle queries, and React client state.
- **Vitest coverage is now broad.** Add/update route tests and orchestrator tests for new auth branches, pruning behavior, stale threshold, and data-health/manual sync UI logic where feasible.

### Integration Points

- **Pruning helper:** likely a new helper in `src/lib/sync/` called by `runFullSync` after the success update. It should return structured counts/errors for metadata.
- **Manual sync auth branch:** preserve GET/POST cron behavior for Vercel. For browser POST with no valid cron secret, check `auth()` and require an authenticated admin session before running sync.
- **Stale banner data source:** either reuse `/api/data-health` or add a smaller authenticated endpoint if the planner wants to avoid fetching full issue tables on every `/search` load. Keep the UX behavior locked; implementation is planner's call.

</code_context>

<specifics>
## Specific Ideas

- The user explicitly rejected external notification integration because the organization does not use that class of tool. Do not reintroduce an external notification provider during planning.
- `/data-health` is the operational source of truth for failed syncs.
- The app-level banner is for genuinely stale data only, not every failed sync. A failure shortly after a successful sync should not scare admins if the active data is still fresh.
- Manual sync should feel deliberate: confirmation first, one request at a time, clear result, then refreshed health cards/history.
- Pruning is cleanup after success, not part of deciding whether the data sync succeeded.

</specifics>
