# Phase 8.7: Operational Maturity - Research

**Researched:** 2026-05-05
**Status:** Ready for planning

## Scope Summary

Phase 8.7 is an operational hardening phase over the already-fixed/tested Phase 8.5/8.6 baseline. It must implement OPS-01..07 without adding external notification providers.

Primary outcomes:
- prune old snapshot-scoped data while preserving active snapshot and `past_session_blocks`
- surface failed/stale sync state in-app through `/data-health`, app-level stale banner, and manual sync feedback
- raise stale warning threshold from 35 minutes to 26 hours
- add authenticated manual sync from `/data-health`
- clean dependency declarations and exact-pin risky packages
- remove small low-risk TODO/dead parameter issues

## Key Findings

### Next.js Route Handler Constraints

Relevant docs read:
- `node_modules/next/dist/docs/01-app/01-getting-started/15-route-handlers.md`
- `node_modules/next/dist/docs/01-app/03-api-reference/04-functions/revalidateTag.md`

Findings:
- App Router route handlers live in `app/**/route.ts` and are request-time for uncached/runtime data. `/api/internal/sync-wise` is a correct place to keep sync triggering.
- Other HTTP methods are not cached by default, so browser `POST` manual sync does not need cache opt-out.
- `revalidateTag` is valid in route handlers. Existing `revalidateTag("snapshot", { expire: 0 })` is acceptable when immediate expiration is required after sync.
- Do not use a Client Component to call `revalidateTag`; keep all cache invalidation in route/server code.

### Snapshot Pruning

Current schema facts:
- Snapshot-scoped tables:
  - `snapshot_stats`
  - `data_issues`
  - `future_session_blocks`
  - `dated_leaves`
  - `recurring_availability_windows`
  - `raw_teacher_tags`
  - `subject_level_qualifications`
  - `tutors`
  - `tutor_identity_group_members`
  - `tutor_identity_groups`
  - `sync_runs` has nullable `snapshotId` / `promotedSnapshotId` references to snapshots
  - `snapshots`
- Cross-snapshot table:
  - `past_session_blocks` is intentionally not snapshot-scoped and must not be deleted.

Because existing FKs do not specify `ON DELETE CASCADE`, deletion should be manually ordered. `sync_runs.snapshot_id` and `sync_runs.promoted_snapshot_id` reference snapshots, so pruning snapshots also needs either:
- set those columns to `null` for pruned snapshots before deleting, or
- exclude snapshots still referenced by sync runs.

Recommended plan direction:
- add helper in `src/lib/sync/snapshot-pruning.ts`
- select snapshots ordered by `createdAt DESC`
- protect latest 30 plus active snapshot id
- compute prune ids from unprotected snapshots
- set `sync_runs.snapshot_id` / `promoted_snapshot_id` to null where they reference prune ids
- delete child rows by `snapshotId` from the snapshot-scoped child tables
- delete `snapshots` by id
- do not touch `pastSessionBlocks`
- return structured result with `attempted`, `deletedSnapshots`, and table row counts if practical

Pruning should run after the existing sync success update in `runFullSync`. If pruning fails, catch/log and update the same successful sync run metadata with pruning failure details. Do not turn the sync result into failure.

### In-App Failure Visibility

The user explicitly rejected external notification integrations. Planning must not add external notification providers or notification env vars.

`/api/data-health/route.ts` already returns:
- `lastSuccessfulSync`
- `lastFailedSync`
- `lastFailureError`
- `staleAgeMs`
- `staleMinutes`
- `activeSnapshotId`
- `recentSyncs`

`/data-health` already displays last successful/failed sync and recent sync history. The manual sync flow should refresh this data after completion.

### Stale Banner

Locked decision:
- app-level banner under shared nav
- visible on `/search` and `/compare`
- trigger only when `staleAgeMs > 48h`
- dismiss per browser session using `sessionStorage`
- plain copy:
  - `Tutor data may be outdated. Last successful sync was over 48 hours ago.`
  - link: `View data health` -> `/data-health`

Implementation options:
- small client component rendered from `src/app/(app)/layout.tsx`
- fetch existing `/api/data-health` and use only `staleAgeMs`
- or create a small authenticated health-summary endpoint to avoid loading issue tables on every workspace load

Planning recommendation: use a small endpoint only if the planner wants to avoid full `/api/data-health` cost; otherwise reuse existing endpoint for lower scope. The user decision is UX behavior, not endpoint shape.

### Stale Threshold Raise

Hard-coded 35-minute locations found:
- `src/lib/search/engine.ts`
- `src/app/api/compare/route.ts`
- `src/app/api/compare/discover/route.ts`

Required target:
- 26 hours = `26 * 60 * 60 * 1000`
- comment must state daily Hobby cron + 2h grace

Existing stale badges/warnings in:
- `src/components/search/search-results.tsx`
- `src/components/search/results-view.tsx`
- `src/components/compare/compare-panel.tsx`

These local indicators can remain but should now use the 26h API threshold. The app-level banner is stricter at 48h.

### Manual Sync Auth

Current route:
- `src/app/api/internal/sync-wise/route.ts`
- GET and POST both use `CRON_SECRET` only
- `src/middleware.ts` bypasses `/api/internal/*`, so internal routes must self-authenticate

Required behavior:
- GET remains cron-secret only
- POST accepts either valid cron secret or authenticated admin session cookie
- browser must never see `CRON_SECRET`
- unauthenticated browser POST returns 401

Auth surface:
- `src/lib/auth.ts` exports `auth()`
- existing route tests mock auth in API tests

Test route with:
- POST valid cron secret -> success
- GET valid cron secret -> success
- POST no/invalid cron secret + valid session -> success
- POST no/invalid cron secret + no session -> 401
- GET no/invalid cron secret -> 401
- failure result returns 500 and no `revalidateTag`

2. **Sync status visibility + stale threshold/banner**
   - covers OPS-02, OPS-03, OPS-04
   - includes app-level banner and threshold constants/tests

3. **Manual sync now**
   - covers OPS-05
   - route auth branch + `/data-health` UI button/feedback/tests

4. **Dependency cleanup and version pinning**
   - covers OPS-06 and package part of OPS-07
   - package/package-lock only plus verification

5. **Small code hygiene**
   - covers `detectConflicts` unused parameter and medium-tier TODO in OPS-07
   - focused compare/modality tests

Wave guidance:
- Wave 1: plan 1 can start first because it touches sync/orchestrator and DB behavior.
- Wave 2: plans 2, 3, 5 can run after or in parallel if planner partitions files carefully. Plan 2 and 3 both touch `/data-health`; avoid assigning them concurrently unless write sets are disjoint.
- Wave 3: dependency cleanup should run after code changes if package-lock churn risks conflicts, or separately if executor sequencing is serial.

### Residual Risks

- Manual sync can approach Vercel's 300s function limit. Do not add retries or long polling beyond one request.
- App-level banner using full `/api/data-health` may query issue tables on every workspace page load. If that becomes heavy, create a small summary endpoint.
- Package lock updates can be noisy; plan should isolate them in one plan.
