# BeGifted Ops — Handbook Overview

BeGifted Ops (historically "BGScheduler", still the production hostname at `bgscheduler.vercel.app`) is the internal admin console that BeGifted Education's non-technical staff use to run their tutoring operation on top of the [Wise](https://api.wiseapp.live) scheduling platform. Wise (tenant `begifted-education`, institute `696e1f4d90102225641cc413`) is the single source of truth for tutors, sessions, leaves, and money; this application periodically pulls that data into versioned Postgres snapshots, normalizes it through a fail-closed pipeline, and serves it back through purpose-built workspaces for searching tutors, comparing schedules, assigning classrooms, chasing renewals, reconciling payroll, auditing activity, and more. Every user-facing page sits behind a Google-OAuth admin gate (`src/middleware.ts`), and almost nothing writes back to Wise without an explicit human click.

The fourteen features below are reachable from the top navigation (`src/components/layout/app-nav.tsx`), grouped into a "Scheduling Tools" menu and a row of operational dashboards. Each maturity badge is taken from the feature's own detail page; follow the link for the rules, flows, and the "why" behind each one.

## Features

**Tutor Search** — *stable (live primary entry point)*. Answers "which tutors are free for a class at this time, and are they qualified for it?" An admin picks a day or date, a time window, a class duration, and optional subject/curriculum/level/modality filters, and gets back a grid of qualified tutors with each candidate sub-slot marked available or blocked, plus a fail-closed "Needs Review" section. It is the application's home page — `/` redirects to `/search` (`src/app/page.tsx`) — and all of its logic runs against the in-memory snapshot index rather than the database. See [docs/features/tutor-search.md](../features/tutor-search.md).

**Tutor Compare** — *legacy-redirect*. Places 1–3 tutors side by side for a chosen week to show what each is already teaching, whether any two share a booking conflict for the same student, and when all selected tutors are simultaneously free. The standalone `/compare` route is now a thin client-side redirect into the search workspace (`src/app/(app)/compare/page.tsx`), but the compare *engine* and its API endpoints are fully active. See [docs/features/tutor-compare.md](../features/tutor-compare.md).

**Sales Dashboard** — *stable*. Imports monthly sales records from Google Sheets, normalizes them into Postgres, and renders a GM-facing command center that tracks revenue pace against a monthly target, scores the trial→new→renewal pipeline, and compares actuals against an imported Bear/Base/Bull projection. It is read-only over the source sheets and owns the repo's shared Google-Sheets access layer (`src/lib/sales-dashboard/sheets.ts`). See [docs/features/sales-dashboard.md](../features/sales-dashboard.md).

**Credit Control** — *stable*. The renewals workspace that keeps every active student's prepaid credit balance from running out unnoticed. It projects when each credit package will fall below an alert threshold (and when it hits zero), ranks at-risk students into a per-admin follow-up queue, and lets admins record outreach status (`contacted` / `pending-callback` / `resolved`). It is read-mostly against Wise — the only persisted state it owns is follow-up status, admin-ownership assignments, and inactivity suppressions. See [docs/features/credit-control.md](../features/credit-control.md).

**Payroll** — *stable*. Reconciles tutor pay against Wise for a chosen Asia/Bangkok calendar month. It pulls every ended teaching session and every payout invoice, joins them to the active tutor-identity snapshot and each teacher's tier, aggregates paid hours and amounts per tutor, and surfaces data-integrity issues (missing/orphan invoices, tier gaps, duration mismatches, expected-rate variances) for an admin to review, adjust, and approve. See [docs/features/payroll.md](../features/payroll.md).

**Wise Activity Audit** — *stable*. A read-only, filterable timeline of operational and financial events inside Wise (session create/update/cancel/delete, billing, invoices, payouts, user and classroom changes) with full raw payloads for forensics, plus a package-sales reconciliation workbench that scores Wise receipt transactions against Sales-Dashboard rows and surfaces a sheet-vs-Wise revenue variance. See [docs/features/wise-activity-audit.md](../features/wise-activity-audit.md).

**Classroom Assignments** — *stable*. Turns each day's Wise teaching sessions into a concrete physical-room plan for the center, then optionally writes the chosen room back to Wise. For a Bangkok date it runs a deterministic engine that respects capacity, TV, online/onsite modality, per-tutor preferred rooms, and tutor continuity, produces a reviewable `(session → room)` table with overrides, publishes eligible OFFLINE rooms on confirmation, and emails each tutor their room route. A nightly cron automates the full sync→assign→publish→email pipeline. See [docs/features/classroom-assignments.md](../features/classroom-assignments.md).

**LINE Integration** — *stable (write-path dry-run only)*. Turns BeGifted's LINE Official Account into a managed scheduling inbox: it ingests every parent message, classifies whether it is a scheduling request, drafts a proposed reply, and surfaces the conversation in a review workspace. Nothing is sent to a parent and no Wise session is mutated without an explicit human click — and the Wise scheduler write-path remains flag-gated and dry-run only. Also covers contact/alias resolution and student linking. See [docs/features/line-integration.md](../features/line-integration.md).

**Room Capacity** — *partial*. Answers "how hard are our rooms being used?" (utilization) and "when will demand outgrow our rooms and tutors?" (forecast). The live utilization path is fully wired — a backfilled history of Wise sessions mapped to normalized rooms, aggregated per day/month/room against a fixed 07:00–21:00 window — while the month-pressure and saturation/breakpoint forecast engines are complete and tested but not yet wired to a frontend. See [docs/features/room-capacity.md](../features/room-capacity.md).

**Data Health** — *stable*. The admin-facing observability dashboard for the Wise snapshot sync pipeline. It answers "is the tutor data the rest of the app searches against fresh and trustworthy right now?" by surfacing sync status and history, snapshot stats, and unresolved normalization issues (unmapped aliases, unresolved modality, unmapped tags) by type and severity, all read from the active snapshot. See [docs/features/data-health.md](../features/data-health.md).

**Tutor Profiles** — *stable*. The editorial layer holding business context Wise does not store — a parent-safe summary, education and language history, English proficiency, young-learner fit, teaching-style and strength tags, curriculum experience, internal use / do-not-use guidance, and a verification trail. Profiles are keyed to a tutor's stable `canonicalKey` so they survive snapshot rebuilds, and admins can edit them directly or bulk-seed them from spreadsheets. See [docs/features/tutor-profiles.md](../features/tutor-profiles.md).

**AI Scheduler** — *experimental*. An LLM-backed assistant that reads pasted parent/admin chat, extracts a structured scheduling request into strict JSON, runs the existing Wise-backed availability search deterministically, and drafts a parent-ready reply — never deciding availability itself ("the model parses, the application decides"). It inherits the same fail-closed Needs-Review and cancelled-session guarantees as manual search. This is the newest, least settled feature; its prompt and evaluation harness are still being iterated. See [docs/features/ai-scheduler.md](../features/ai-scheduler.md).

**Proposals (Admin Holds)** — *experimental*. Lets admin staff temporarily "hold" one or more tutor slots while a deal with a parent is still being negotiated, recording who holds it, for which student, and on what day/time, then surfacing that hold inside the availability search so a second admin will not offer the same slot to a different parent. Holds are deliberately local advisory overlays — never written back to Wise — and are reached from the search workspace rather than a standalone page. See [docs/features/proposals.md](../features/proposals.md).

**Leave Requests** — *in progress (uncommitted)*. The admin workspace for triaging tutor time-off submissions. Tutors fill out a Google Form whose responses land in a Sheet; the feature pulls those rows into Postgres, matches each to a Wise tutor identity, computes which upcoming Wise sessions overlap the requested leave window, and gives admins a worklist to review, set a workflow status, leave a staff note, and optionally write a status string back to the source sheet. The migration, schema, module, routes, page, nav link, and cron all exist in the working tree but are uncommitted at this revision. See [docs/features/leave-requests.md](../features/leave-requests.md).

## System scale

At this revision the application comprises **78 database tables** (`pgTable` definitions in `src/lib/db/schema.ts`), **110 HTTP endpoints** (method exports across `src/app/api/**/route.ts`), and **14 admin pages** (the `page.tsx` files under the `src/app/(app)/` route group). For the mechanical inventory behind these numbers, see the [API reference index](../reference/api/index.md) and the [database reference index](../reference/database/index.md).
