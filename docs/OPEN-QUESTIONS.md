## Sync, concurrency & snapshots

- **Cold-build latency on Vercel is unmeasured against the "<400ms warm" target.** `SearchIndex` lives per serverless instance on `globalThis`; cold instances pay a full `buildIndex` on first request. How often instances recycle/scale out â€” and whether cold starts hit users after a deploy â€” is unknown. Worth measuring.
- **A brand-new environment surfaces an error on every read until the first sync.** `buildIndex` throws "No active snapshot found" (`index.ts:151`) and `ensureIndex` only swallows the no-active case when a cached index already exists (`index.ts:384`); `getActiveSnapshotIdOrThrow` throws the same. An environment that has never completed a sync would error on every read route until the first promotion. Do deploy runbooks guarantee a promoted snapshot before serving traffic?
- **Per-request DB round-trip on the "warm" path.** `getTutorProfileVersion` runs a `count(*)` + `max(updated_at)` aggregate on `tutor_business_profiles` on **every** `ensureIndex` call (`index.ts:368-375`), including the warm path â€” a per-request DB hit the "zero DB on hot path" framing glosses over. Confirm the table stays small.
- **Overlap-guard semantics differ across the syncs.** Wise snapshot uses a DB single-flight row + 20-min sweep; `wise-activity` and `leave-requests` use an in-process `AlreadyRunningError` â†’ HTTP 409; but the `sales-dashboard` and `credit-control` route handlers show no overlap guard. Is concurrency protection handled inside `importRefreshableSalesSources` / `runCreditControlSyncRequest`, or can those two run concurrently?
- **The 20-minute abandoned-`running` cutoff is wider than both the 800s function ceiling and (parts of) the cron interval.** `run-wise-sync.ts:10` sets a 20-min stale cutoff vs. the sync's `maxDuration = 800` (~13.3 min) ceiling (`sync-wise/route.ts:6`); a wedged `running` row blocks ~1.5 cron cycles before cleanup. Internally consistent as a safety net today, but flagged if the cron cadence tightens.
- **No first-class re-promote / rollback endpoint.** Rollback is implicit (re-run the sync from the last good snapshot); `pruneOldSnapshots` hard-deletes everything beyond the most recent 30 (`SNAPSHOT_RETENTION_COUNT`). Should a supported manual-rollback path exist, and what is the approved DB-level procedure?
- **`/api/admin/sync-wise` exists alongside `/api/internal/sync-wise`.** The classroom `sync-flow.ts:110` posts to `/api/admin/sync-wise`. The route **does** exist (`src/app/api/admin/sync-wise/route.ts`, with a test), so its existence is settled â€” the open question is narrower: is `/api/admin/sync-wise` the intended canonical *UI-facing* sync entry point, vs. `/api/internal/sync-wise`?
- **Auth-logic duplication across sync routes.** `cron-auth.ts` provides `rejectInvalidCronSecret`, but `sync-wise`, `sync-sales-dashboard`, `sync-credit-control`, and `sync-room-utilization` each re-implement an inline `hasValidCronSecret` with identical constant-time logic instead of importing the helper. Worth consolidating?
- **Dual-auth labelling on internal sync routes.** GET handlers for `sync-wise`, `sync-credit-control`, `sync-sales-dashboard` are cron-only (`allowSessionAuth:false`), while their POST handlers also accept an admin-session fallback; `sync-room-utilization` has only POST with the session fallback. Confirm the "cron (or admin on POST)" presentation is acceptable for the API index vs. deferring detail to `internal-crons.md`.
